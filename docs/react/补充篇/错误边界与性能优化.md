React 的高级能力里，有两类内容在业务中最容易直接见效：一类是出错时兜底，另一类是避免无意义渲染。

## 错误边界

错误边界（Error Boundary）用于捕获后代组件在==渲染阶段、生命周期、构造函数==里抛出的错误，并展示降级 UI。

::: table title="错误边界能抓什么" full-width
| 能捕获 | 不能捕获 |
| --- | --- |
| render 抛错 | 事件处理函数中的异常 |
| 生命周期抛错 | `setTimeout` / Promise 等异步错误 |
| 子树初始化异常 | 服务端渲染过程中的异常 |
:::

:::code-tabs
@tab ErrorBoundary.jsx
```jsx
import React from 'react';

export default class ErrorBoundary extends React.Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError() {
    // 一旦子组件抛错，就切到降级界面
    return { hasError: true };
  }

  componentDidCatch(error, info) {
    // 可以在这里接入日志平台
    console.error('组件渲染出错：', error, info);
  }

  render() {
    if (this.state.hasError) {
      return <div>页面局部异常，请稍后重试。</div>;
    }

    return this.props.children;
  }
}
```

@tab BuggyPanel.jsx
```jsx
export default function BuggyPanel({ crashed }) {
  if (crashed) {
    throw new Error('模拟渲染期错误');
  }

  return <div>当前组件渲染正常</div>;
}
```

@tab App.jsx
```jsx
import { useState } from 'react';
import BuggyPanel from './BuggyPanel';
import ErrorBoundary from './ErrorBoundary';

export default function App() {
  const [crashed, setCrashed] = useState(false);

  return (
    <div>
      <button onClick={() => setCrashed(true)}>触发错误</button>

      <ErrorBoundary>
        <BuggyPanel crashed={crashed} />
      </ErrorBoundary>
    </div>
  );
}
```

## 先做结构性优化，再做缓存优化

很多性能问题并不是“少写一个 `useMemo`”就能解决，而是组件边界和状态位置一开始就不合理。

::: table title="更值得优先排查的优化点" full-width
| 优先级 | 该先看什么 | 原因 |
| --- | --- | --- |
| 高 | 状态是否放得过高 | 上层状态一变，整片子树都会跟着算 |
| 高 | 组件是否拆得过粗 | 粗组件更容易把无关更新带下去 |
| 高 | 列表 `key` 是否稳定 | 不稳定 `key` 会让复用失效 |
| 中 | Context `value` 是否频繁新建 | 会唤醒所有消费方参与计算 |
| 中 | 是否真的存在昂贵计算 | 没有成本，就没有缓存收益 |
:::

## 类组件时代的优化方式

课程里也讲了 `shouldComponentUpdate` 和 `PureComponent`。现在虽然函数组件更主流，但读旧代码仍然会遇到它们。

::: table title="旧 API 与现代 API 的对应关系" full-width
| 类组件方案 | 现代函数组件里的对应思路 |
| --- | --- |
| `shouldComponentUpdate` | `React.memo` + 自定义比较 |
| `PureComponent` | `React.memo` |
| 手动比较派生值 | 派生值缓存 |
:::

## 性能优化不是越多越好

:::warning
- `memo` 有比较成本，组件本身很轻时，强上优化反而可能更慢。
- 缓存类 API 不是“写了就快”，只有在稳定引用确实能减少后续渲染时才有价值。
- 真正重要的是保持不可变更新、合理拆组件、减少不必要的上层状态。
:::

## 一份更实用的优化清单

:::collapse
- 先确认有没有真实性能问题，再决定是否加优化
- 把高频变化的状态放到更靠近消费它的组件
- 给列表项提供稳定 `key`
- 避免在 render 里创建无意义的新对象、新数组、新函数并层层下传
- 对计算昂贵的派生值，再考虑缓存式优化
- 对依赖稳定引用的场景，再考虑记忆化策略
- 对纯展示型子组件优先考虑 `React.memo`
:::
