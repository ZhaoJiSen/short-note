## Update 是 React 内部统一的“更新单元”

不管更新来自 `setState`、`dispatch(action)`，还是别的内部流程，React 最后都会把它抽象成一个 `Update` 对象。

```js
const update = {
  lane,          // 本次更新的优先级
  action,        // 对状态 Hook 来说，就是 setState 传入的值或函数
  hasEagerState: false,
  eagerState: null,
  next: null,
};
```

这一层抽象很关键，因为它让 React 可以用同一套调度与合并逻辑来处理不同来源的更新。

## updateQueue：多个更新怎么排队

单个 Update 不会直接生效，它会先进入 `updateQueue`。

```js
const updateQueue = {
  baseState: null,
  firstBaseUpdate: null,
  lastBaseUpdate: null,
  shared: {
    pending: null, // 新 update 先在这里形成环状链表
  },
};
```

计算状态时，React 大致会做两件事：

1. 把 `shared.pending` 里的新 update 拼接到 base update 链表后面
2. 按当前 render 的 `lanes` 挑出可执行 update，基于 `baseState` 依次计算出新的状态

::: table title="为什么需要队列而不是直接覆盖" full-width
| 需求 | 如果没有队列会怎样 |
| --- | --- |
| 同一事件里连续触发多次更新 | 后面的更新可能直接覆盖前面的意图 |
| 不同优先级更新并存 | React 无法跳过低优先级，先处理高优先级 |
| 更新可能被打断后恢复 | React 难以从中间状态继续算下去 |
:::

## eagerState 与 bailout：React 怎么少做一些无用功

### eagerState

在某些简单场景下，React 会尝试提前算出“这次更新后的状态是不是和当前一样”。如果答案是“没变”，就有机会提前结束，不必真的走完整个调度流程。

### bailout

当 React 发现 props、state、context 等输入没有影响当前节点时，它就可能跳过当前 Fiber 及其部分子树的继续计算，这就是 bailout。

::: table title="两种优化策略的区别" full-width
| 策略 | 优化位置 | 目标 |
| --- | --- | --- |
| `eagerState` | 更新入队前后 | 尽早发现“其实状态没变” |
| `bailout` | render 过程中 | 跳过没必要继续处理的 Fiber 子树 |
:::

## 为什么 Context 容易影响优化

`Context` 的问题不在于“不能用”，而在于 Provider 的 `value` 一变化，消费它的组件就会重新参与计算。哪怕父组件已经能 bailout，依赖上下文的消费方依旧可能被唤醒。

```jsx
const value = { theme, setTheme }; // 每次 render 都是新对象

<ThemeContext.Provider value={value}>
  {children}
</ThemeContext.Provider>
```

所以 Context 经常要配合稳定引用一起使用，否则 Provider 每次 render 都可能让优化失效。

## 对日常开发的真实启示

:::collapse
- 不要在组件里随手造新对象再往下传，尤其是 Provider 的 `value`
- 需要基于旧值计算新值时，用函数式更新
- 真正高收益的优化通常不是“补一个 memo”，而是减少无关更新的传播范围
- 如果一个状态放得太高，再精细的缓存也只能事后补救
- 理解 `Update -> 调度 -> render -> bailout` 这条链路，比死记某个 API 更重要
:::
