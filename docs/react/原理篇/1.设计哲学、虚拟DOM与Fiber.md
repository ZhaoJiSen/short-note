这一章把课程里“前端框架的理解”“React 和 Vue 描述页面的区别”“虚拟 DOM”“React 整体架构”的前置部分合并起来。先把设计目标讲清楚，再去看后面的 render、commit、Hook 内部原理。

## React 是库还是框架

严格来说，React 核心提供的是：

1. 基于状态的声明式渲染
2. 组件化开发能力

但一个现代前端项目显然还需要路由、状态管理、构建、部署、SSR 等能力，所以“React 技术栈”才更接近完整框架。

::: table title="UI 库与完整框架" full-width
| 维度 | React 核心 | React 生态 / 框架 |
| --- | --- | --- |
| 关注点 | 渲染 UI | 路由、数据获取、SSR、工程化 |
| 典型包 | `react`、`react-dom` | React Router、Redux Toolkit、Next.js |
| 本质角色 | UI Library | Solution / Framework |
:::

## JSX 与模板，代表两种 UI 描述思路

React 倾向于用 `JSX` 描述界面，Vue 更常见的是模板。两者都能生成 UI，但起点不一样。

::: table title="React 与 Vue 的描述方式" full-width
| 方案 | 出发点 | 典型特征 |
| --- | --- | --- |
| JSX | 扩展 JS，让逻辑也能描述 UI | 灵活、表达式能力强 |
| 模板 | 扩展 HTML，让 UI 中也能描述逻辑 | 结构直观、模板感更强 |
:::

React 团队的判断是：==UI 和逻辑本来就会互相引用，比如事件绑定、条件渲染、列表渲染，所以直接扩展 JS 更顺手==。

## 什么是虚拟 DOM

`VDOM` 不是某个具体对象的名字，而是一种思想：==在内存里先用理想化的数据结构描述 UI，再把变更同步到真实宿主环境==。

在 React 里，JSX 最终会得到一个 React Element 对象：

:::code-tabs
@tab JSX
```jsx
const element = <h1 className="title">Hello React</h1>;
```

@tab 编译后（简化）
```js
import { jsx } from 'react/jsx-runtime';

const element = jsx('h1', {
  className: 'title',
  children: 'Hello React',
});
```

@tab ReactElement 结构（简化）
```js
const element = {
  type: 'h1',
  key: null,
  ref: null,
  props: {
    className: 'title',
    children: 'Hello React',
  },
};
```
:::

> [!NOTE]
> 所以“虚拟 DOM 是一个 JS 对象”这句话只说对了一半。更准确的说法是：虚拟 DOM 是一种思想，React 用 JS 对象实现了这种思想。

## 为什么需要虚拟 DOM

虚拟 DOM 的价值主要体现在两个方面：

::: table title="虚拟 DOM 的两个核心价值" full-width
| 价值 | 说明 |
| --- | --- |
| 更新阶段更容易做最小化变更 | 先在 JS 层比较差异，再决定哪些真实节点要改 |
| 提供跨平台抽象 | 同一套 ReactElement/Fiber，可对接浏览器、Native、测试环境等不同渲染器 |
:::

:::details 一个经常被误解的点
虚拟 DOM 不等于“首次渲染一定更快”。它的优势主要体现在==更新阶段==，尤其是在复杂 UI 频繁变更时，更容易做到最小成本的真实 DOM 修改。
:::

## Fiber：从“描述结果”到“描述工作过程”

React Element 更像“我要什么 UI”，而 Fiber 则进一步记录了“这份 UI 在更新过程中需要哪些额外信息”。

简化后的 Fiber 节点可以这样理解：

```js
function FiberNode(type, pendingProps, key) {
  this.type = type;
  this.key = key;

  // Fiber 树的链表关系
  this.return = null;   // 指向父节点
  this.child = null;    // 指向第一个子节点
  this.sibling = null;  // 指向下一个兄弟节点

  // 当前节点对应的输入与输出
  this.pendingProps = pendingProps;
  this.memoizedProps = null;
  this.memoizedState = null;

  // 更新、副作用、优先级
  this.updateQueue = null;
  this.flags = 0;
  this.lanes = 0;

  // current / workInProgress 双缓冲
  this.alternate = null;
}
```

:::table title="ReactElement 与 Fiber 的区别" full-width
| 对象 | 更像什么 | 关注点 |
| --- | --- | --- |
| ReactElement | UI 描述结果 | 标签类型、props、children |
| Fiber | 一份可调度的工作单元 | 父子兄弟关系、状态、更新队列、优先级、副作用 |
:::

## Fiber 为什么能解决旧架构问题

React 15 之前的 Stack 架构，本质上更像“递归到底，一口气做完”。这会带来两个问题：

1. 如果组件树很大，递归比较过程不可中断，JS 会长时间占用主线程。
2. 所有更新缺乏细粒度优先级，输入、动画、普通列表刷新会抢同一条路。

Fiber 的关键变化在于：

- 用链表代替递归式调用栈
- 每个 Fiber 都可以被看成一个可暂停、可恢复的工作单元
- 可以把“调度”和“计算 UI”拆开

:::collapse
- 一句话理解 Fiber

  - ReactElement 解决“界面该长什么样”
  - Fiber 解决“界面更新工作该怎么被拆开、暂停、恢复、提交”
:::
