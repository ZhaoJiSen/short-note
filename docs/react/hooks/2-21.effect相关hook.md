[+Effect]: 渲染结束后，用来同步外部系统的副作用
[+Cleanup]: effect 返回的清理函数，用于解绑、取消订阅、销毁定时器
[+Paint]: 浏览器把最新布局真正绘制到屏幕的过程

`useEffect` / `useLayoutEffect` 处理的不是“界面长什么样”，而是==界面渲染出来之后，要不要去同步浏览器、网络、订阅、计时器这些外部世界==。

::: table title="effect 相关 Hook 的分工" full-width
| Hook | 执行时机 | 适合场景 |
| --- | --- | --- |
| `useEffect` | 浏览器完成 Paint[+Paint] 之后 | 请求数据、订阅事件、启动定时器 |
| `useLayoutEffect` | DOM 提交后、浏览器绘制前 | 读布局、同步滚动、避免闪动 |
:::

## useEffect：大部分副作用都从这里开始

一个标准的 `useEffect` 通常包含三件事：建立副作用、响应依赖变化、在离开时清理副作用。

:::code-tabs
@tab ResizeReporter.jsx
```jsx
import { useEffect, useState } from 'react';

function getWindowSize() {
  return {
    width: window.innerWidth,
    height: window.innerHeight,
  };
}

export default function ResizeReporter() {
  const [size, setSize] = useState(() => getWindowSize());

  useEffect(() => {
    function handleResize() {
      // 窗口尺寸变化后，更新组件状态
      setSize(getWindowSize());
    }

    window.addEventListener('resize', handleResize);

    return () => {
      // 组件卸载前，清理订阅，避免内存泄漏
      window.removeEventListener('resize', handleResize);
    };
  }, []);

  return (
    <section>
      <h3>当前窗口尺寸</h3>
      <p>
        {size.width}px × {size.height}px
      </p>
    </section>
  );
}
```

@tab App.jsx
```jsx
import ResizeReporter from './ResizeReporter';

export default function App() {
  return (
    <main>
      <h2>useEffect 示例</h2>
      <ResizeReporter />
    </main>
  );
}
```
:::

## 依赖数组到底在表达什么

依赖数组不是“语法装饰”，而是在告诉 React：==这个副作用依赖哪些值，只要它们没变，就不必重新执行==。

::: table title="三种依赖写法" full-width
| 写法 | 含义 | 常见场景 |
| --- | --- | --- |
| 不传第二个参数 | 每次 render 后都执行 | 极少数必须每次同步的副作用 |
| `[]` | 只在挂载后执行一次，卸载时清理 | 事件绑定、一次性初始化 |
| `[roomId, keyword]` | 首次执行，依赖变化时重跑 | 按参数请求数据、重建订阅 |
:::

:::warning
不要把“根据 props/state 计算一个新值”写进 `useEffect` 再 `setState` 回去。那不是副作用，而是派生值，通常更应该用 `useMemo` 或直接在 render 中计算。
:::

## useLayoutEffect：只在需要“先读布局、再绘制”时使用

`useLayoutEffect` 会比 `useEffect` 更早执行。它适合那些一旦晚一拍就会看到闪动的场景，比如测量 DOM、立即滚动到指定位置、同步选中文本。

:::code-tabs
@tab AutoFocusCard.jsx
```jsx
import { useLayoutEffect, useRef, useState } from 'react';

export default function AutoFocusCard() {
  const inputRef = useRef(null);
  const [rect, setRect] = useState(null);

  useLayoutEffect(() => {
    const element = inputRef.current;
    if (!element) return;

    // 在浏览器绘制前读取布局并处理焦点
    element.focus();
    setRect(element.getBoundingClientRect());
  }, []);

  return (
    <section>
      <input ref={inputRef} placeholder="页面加载后会自动聚焦" />

      {rect ? (
        <p>
          输入框宽度：{Math.round(rect.width)}px，高度：{Math.round(rect.height)}px
        </p>
      ) : null}
    </section>
  );
}
```

@tab App.jsx
```jsx
import AutoFocusCard from './AutoFocusCard';

export default function App() {
  return (
    <main>
      <h2>useLayoutEffect 示例</h2>
      <AutoFocusCard />
    </main>
  );
}
```
:::

::: card title="什么时候不要上 useLayoutEffect" icon="material-icon-theme:warning"
如果只是发请求、打日志、绑事件、开定时器，用 `useEffect` 就够了。`useLayoutEffect` 会阻塞浏览器绘制，用重了会让页面更卡。
:::

## React 内部怎么看待 effect

从源码视角看，effect hooks 不只是“执行一个回调”，它们会经历声明、调度、执行三个阶段。

::: table title="effect 的内部阶段" full-width
| 阶段 | React 在做什么 |
| --- | --- |
| 声明阶段 | render 时把 effect 信息挂到当前 Fiber 上 |
| 调度阶段 | 把本次需要执行的 effect 标记进 commit 流程 |
| 执行阶段 | 在 commit 对应时机运行回调和清理函数 |
:::

这也是为什么 `useEffect` 写在函数体里，但真正执行是在 DOM 更新之后。

## 常见误区

:::collapse
- 缺依赖不是优化手段，它更常见的结果是闭包拿到旧值
- effect 里发起的异步任务，最好有取消标记或中断逻辑
- 需要清理的副作用，一定返回 cleanup 函数
- 不要把 effect 当“生命周期翻译器”，它更像“与外部系统同步”
- 能在 render 里直接算出来的值，不要绕一圈放到 effect 里
:::

> [!NOTE]
> `useInsertionEffect` 也属于 effect 家族，但更偏 CSS-in-JS 库这类底层场景。常规业务开发里，重点掌握 `useEffect` 和 `useLayoutEffect` 就够了。
