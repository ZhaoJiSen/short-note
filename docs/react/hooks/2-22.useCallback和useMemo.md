[+Memoization]: 记忆化，把已有结果缓存起来
[+Bailout]: React 在发现输入没变时，跳过部分计算的优化过程
[+浅比较]: 只比较引用或第一层字段是否变化

`useMemo` 和 `useCallback` 经常一起出现，但它们缓存的不是同一个东西。真正要记住的是：==它们不是性能开关，而是给比较与 Bailout[+Bailout] 创造条件==。

::: table title="两个 Hook 的核心区别" full-width
| Hook | 缓存什么 | 返回什么 | 最常见用途 |
| --- | --- | --- | --- |
| `useMemo` | 计算结果 | 一个值 | 昂贵计算、稳定对象、列表过滤结果 |
| `useCallback` | 函数引用 | 一个函数 | 给 memo 子组件传回调、稳定依赖 |
:::

## 一个完整例子：和 `React.memo` 配合使用

这组 API 最常见的价值，不是“单独用了会快”，而是跟 `React.memo` 配合后，让子组件有机会跳过无意义渲染。

:::code-tabs
@tab App.jsx
```jsx
import { useCallback, useMemo, useState } from 'react';
import TodoList from './TodoList';

export default function App() {
  const [keyword, setKeyword] = useState('');
  const [count, setCount] = useState(0);

  const todos = useMemo(
    () => [
      { id: 1, title: '学习 React' },
      { id: 2, title: '学习 Router' },
      { id: 3, title: '学习 Zustand' },
    ],
    []
  );

  const filteredTodos = useMemo(() => {
    // 过滤结果只在 keyword 或 todos 变化时重算
    return todos.filter((item) => item.title.includes(keyword));
  }, [keyword, todos]);

  const handleSelect = useCallback((todo) => {
    console.log('当前点击：', todo.title);
  }, []);

  return (
    <div>
      <input
        value={keyword}
        onChange={(event) => setKeyword(event.target.value)}
        placeholder="搜索待办"
      />

      <button onClick={() => setCount((prev) => prev + 1)}>
        计数：{count}
      </button>

      <TodoList list={filteredTodos} onSelect={handleSelect} />
    </div>
  );
}
```

@tab TodoList.jsx
```jsx
import React from 'react';

function TodoList({ list, onSelect }) {
  console.log('TodoList render');

  return (
    <ul>
      {list.map((todo) => (
        <li key={todo.id}>
          <button onClick={() => onSelect(todo)}>{todo.title}</button>
        </li>
      ))}
    </ul>
  );
}

// props 浅比较相同时，直接复用上一次渲染结果
export default React.memo(TodoList);
```

## 怎么理解这两个 Hook

::: table title="心智模型" full-width
| API | 可以把它理解成什么 |
| --- | --- |
| `useMemo(fn, deps)` | “依赖不变时，复用上一次算出来的值” |
| `useCallback(fn, deps)` | “依赖不变时，复用上一次的函数引用” |
:::

::: card title="先问三个问题，再决定要不要加缓存" icon="material-icon-theme:speed"
- 这个计算真的昂贵吗
- 这个引用稳定下来之后，后续比较真的能少 render 吗
- 不加缓存时，页面是否真的有肉眼可感知的问题
:::

## 什么时候值得用

`useMemo` 常见在两类地方：

- 计算代价高，比如大列表筛选、复杂格式化、图表数据整理
- 需要把对象或数组引用稳定下来，避免子组件反复收到“新对象”

`useCallback` 常见在两类地方：

- 传给 `React.memo` 子组件的回调
- 某个 effect 依赖函数本身，希望函数引用只在关键依赖变化时更新

## 什么时候不要用

:::warning
- 组件本身很轻时，记忆化比较成本可能比重新 render 还高
- 只是为了“看起来专业”而到处包 `useMemo` / `useCallback`，通常没有收益
- 依赖项写不全会制造陈旧闭包问题，结果比不优化更难查
:::

## 原理视角：React 记住了什么

在 Hook 链表里，`useMemo` 和 `useCallback` 也会占据一个节点。它们的 `memoizedState` 通常会保存“上一次结果 + 上一次依赖”。

```js
// useMemo 的简化理解
hook.memoizedState = [cachedValue, deps];

// useCallback 的简化理解
hook.memoizedState = [cachedFn, deps];
```

React 下一次 render 时，会先比较新的 `deps` 和旧的 `deps`。如果没变，就直接复用之前缓存的值或函数引用。

:::collapse
- `useMemo` 解决“值重复算”的问题
- `useCallback` 解决“函数每次都是新引用”的问题
- 它们的价值通常建立在 `React.memo`、依赖比较、Context value 稳定这些场景上
- 真正更重要的优化，仍然是状态下沉、合理拆组件、避免不必要的上层更新
:::
