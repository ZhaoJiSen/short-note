[+Store]: 全局状态仓库
[+RTK]: Redux Toolkit

当共享状态越来越多、组件层级越来越深时，只靠“状态提升 + props 逐层传递”会开始吃力。这时候就需要把共享状态抽出去，交给统一约定来管理。

## 为什么需要状态管理

::: table title="不同状态的存放位置" full-width
| 状态类型 | 推荐放置位置 | 例子 |
| --- | --- | --- |
| 组件私有状态 | 组件内部 `useState` | 折叠面板开关、输入框值 |
| 少量跨层共享状态 | `Context` | 当前主题、登录用户信息 |
| 中大型业务共享状态 | Redux Toolkit | 用户、订单、权限、缓存状态 |
| 轻量全局共享状态 | Zustand | 面板开关、购物车、筛选条件 |
:::

> [!TIP]
> 状态管理库不是用来“替代所有本地状态”的。局部状态依然应该留在组件内部，只有确实需要共享、追踪、复用时再抽出去。

## Context API：React 自带的共享方案

`Context` 解决的是跨层级传值问题，核心思路是：Provider 提供上下文，后代组件直接消费，不再一层层 props drilling。

```jsx
import { createContext, useContext, useMemo, useState } from 'react';

const ThemeContext = createContext(null);

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');

  // 避免每次 render 都产生新对象，导致消费方全量更新
  const value = useMemo(() => ({ theme, setTheme }), [theme]);

  return (
    <ThemeContext.Provider value={value}>
      {children}
    </ThemeContext.Provider>
  );
}

function ThemeButton() {
  const { theme, setTheme } = useContext(ThemeContext);

  return (
    <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
      当前主题：{theme}
    </button>
  );
}

export default function App() {
  return (
    <ThemeProvider>
      <ThemeButton />
    </ThemeProvider>
  );
}
```

:::details Context 适合什么，不适合什么
- 适合：主题、语言、权限、用户信息这类“低频变化但很多地方要用”的数据
- 不太适合：高频更新的大型业务状态，因为 Provider 的 value 一变，消费组件会跟着重新计算
:::

## Redux Toolkit：强约束、可预测的数据流

Redux 的核心思想是：

1. 单一数据源
2. state 只读
3. 通过 reducer 计算新状态

现代 React 项目里，一般不再手写传统 Redux 四件套，而是直接使用 `RTK + React Redux`。

:::code-tabs
@tab store.js
```js
import { configureStore } from '@reduxjs/toolkit';
import counterReducer from './counterSlice';

export const store = configureStore({
  reducer: {
    counter: counterReducer,
  },
});
```

@tab counterSlice.js
```js
import { createSlice } from '@reduxjs/toolkit';

const counterSlice = createSlice({
  name: 'counter',
  initialState: {
    value: 0,
  },
  reducers: {
    increment(state) {
      // RTK 内部借助 Immer，让“看起来像修改”也能得到不可变更新结果
      state.value += 1;
    },
    addByAmount(state, action) {
      state.value += action.payload;
    },
  },
});

export const { increment, addByAmount } = counterSlice.actions;
export default counterSlice.reducer;
```

@tab main.jsx
```jsx
import React from 'react';
import ReactDOM from 'react-dom/client';
import { Provider } from 'react-redux';
import App from './App';
import { store } from './store';

ReactDOM.createRoot(document.getElementById('root')).render(
  <Provider store={store}>
    <App />
  </Provider>
);
```

@tab App.jsx
```jsx
import { useDispatch, useSelector } from 'react-redux';
import { addByAmount, increment } from './counterSlice';

export default function App() {
  const value = useSelector((state) => state.counter.value);
  const dispatch = useDispatch();

  return (
    <section>
      <h2>Redux Toolkit Counter</h2>
      <p>当前值：{value}</p>
      <button onClick={() => dispatch(increment())}>+1</button>
      <button onClick={() => dispatch(addByAmount(5))}>+5</button>
    </section>
  );
}
```
:::

## Zustand：轻量直接的全局状态

如果你不需要 Redux 那么强的约束，只想快速得到一个可用的全局 store，Zustand 会非常顺手。它的特点是 API 小、样板代码少、上手快。

:::code-tabs
@tab useCounterStore.js
```js
import { create } from 'zustand';

export const useCounterStore = create((set) => ({
  count: 0,
  increment: () =>
    set((state) => ({
      count: state.count + 1,
    })),
  addByAmount: (step) =>
    set((state) => ({
      count: state.count + step,
    })),
}));
```

@tab App.jsx
```jsx
import { useCounterStore } from './useCounterStore';

export default function App() {
  const count = useCounterStore((state) => state.count);
  const increment = useCounterStore((state) => state.increment);
  const addByAmount = useCounterStore((state) => state.addByAmount);

  return (
    <section>
      <h2>Zustand Counter</h2>
      <p>当前值：{count}</p>
      <button onClick={increment}>+1</button>
      <button onClick={() => addByAmount(5)}>+5</button>
    </section>
  );
}
```
:::

## Redux Toolkit 和 Zustand 怎么选

::: table title="Redux Toolkit vs Zustand" full-width
| 维度 | Redux Toolkit | Zustand |
| --- | --- | --- |
| 约束程度 | 强，数据流更统一 | 弱，更自由 |
| 样板代码 | 相对更多，但 RTK 已大幅简化 | 很少 |
| 调试与生态 | DevTools、middleware、团队协作成熟 | 轻量，生态没那么重 |
| 适合场景 | 中大型业务、多人协作、状态演进复杂 | 中小型项目、快速开发、局部全局状态 |
:::

:::collapse
- 一个务实的选择策略

  - 只有少量共享状态：先用 `Context`
  - 需要快速全局共享：优先试 `Zustand`
  - 状态流复杂、需要强约束、团队协作重：选 `Redux Toolkit`
:::
