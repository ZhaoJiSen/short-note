> [!IMPORTANT]
> 渲染器负责将虚拟 DOM 转换为真实 DOM，并将其渲染到页面中

## render 函数的基本实现

::::: steps

1. `render` 函数的基本设计

   `render` 函数就是用于把一个虚拟节点渲染到一个真实容器中，所以 `render` 至少要有两个参数：这一次要渲染的虚拟节点 `vnode`，以及真实的挂载容器 `container`

   ```ts
   function render(vnode, container) {}
   ```

   函数有了，接下来要考虑它的功能实现

   :::info 实现思路
   - 在首次渲染时：容器中还没有旧的 `vnode`，因此这个时候需要做的是挂载，也就是调用 `mountElement` 函数
   - 再次渲染时：容器中已经有了旧的 `vnode`，因此这个时候需要做的是比较新旧 `vnode` 并更新，也就是调用 `patch` 函数
   - 卸载：新的 `vnode` 不存在，但容器里还有旧内容，这时候要把旧内容移除
   :::

   :::code-tabs

   @tab renderer.ts

   ```ts
   function render(vnode, container) {
     if (vnode) {
       patch(container._vnode, vnode, container);
     } else {
       // 此时表明旧的虚拟 DOM 存在，但新的虚拟 DOM 不存在，为 unmount 操作
       if (container._vnode) {
         container.innerHTML = "";
       }
     }

     // 将每次的 vnode 存储到 container 对象上，以便下次更新时作为旧的 vnode 进行比较
     container._vnode = vnode;
   }
   ```

   :::

2. `patch` 函数的基本实现

   在 `render` 函数中，首次渲染和再次渲染都是通过 `patch` 函数来进行处理的，之所以二者都使用该函数是因为 `patch` 函数的设计是为了处理新旧 `vnode` 之间的差异，然后进行渲染操作，而首次渲染完全可以看做一次特殊的更新（旧 `vnode` 不存在），因此可以将逻辑进行整合

   由此 `patch` 函数需要接收三个参数：旧的 `vnode`、新的 `vnode` 和容器

   :::info 实现思路
   - 如果旧的 `vnode` 不存在，说明这是一次首次渲染，此时直接调用 `mountElement` 来挂载新的 `vnode`
   - 如果旧的 `vnode` 存在，那么在此时的 `patch` 中就意味着更新
   :::

   ```ts
   function patch(oldVNode, newVNode, container) {
     if (!oldVNode) {
       mountElement(newVNode, container);
     } else {
       // 更新逻辑
     }
   }
   ```

3. `createRenderer` 函数的设计

   在 `render` 函数的基本实现中，已经不可避免的将一些其他函数的调用写在了 `render` 中。这也表明他们之间是有耦合关系的，所以可以通过工厂函数 `createRenderer` 将逻辑进行整合

   ```ts
   function createRenderer() {
     function render(vnode, container) {}

     function patch() {}

     function unmount() {}

     function mountElement() {}

     return {
       patch,
       render,
       unmount,
       mountElement,
     };
   }
   ```

4. 平台依赖性的拆分

   在 Vue 源码中，渲染器的设计还考虑了平台依赖性的问题。因为 Vue 不仅可以运行在浏览器环境中，还可以运行在其他平台（如服务器端、原生应用等）中，所以将 DOM 操作与渲染逻辑进行了整体的拆分

   渲染逻辑放入 `runtime-core` 中，平台相关实现放入 `runtime-dom` 中。这样一来，渲染器就可以通过传入不同的 "平台能力" 来适配不同的运行环境，而不需要修改核心的渲染逻辑

   在现阶段的实现中，先试用一个全局对象 `options` 来代替 `runtime-dom` 包中的内容

   :::code-tabs
   @tab renderer.ts

   ```ts
   const options = {
     createElement(tag) {
       return document.createElement(tag)
     },
     setElementText(el, text) {
       el.textContent = text
     },
     insert(el, parent, anchor = null) {
       parent.insertBefore(el, anchor)
     }
   }


   function createRenderer(options) {
     const { createElement, insert, setElementText } = options

     function render() {}
     function patch() {}
     function unmount() {}
     function mountElement() {}

     return {
       render,
       patch,
       unmount,
       mountElement
     }
   }


   const renderer = createRenderer(options);
   ```

   :::

## 挂载的基本实现

[+值不同]: 以 `<input value="foo" />` 为例，`getAttribute('value')` 读取的是初始属性值，而 `el.value` 读取的是当前输入框的实时值。用户修改内容后，二者可能并不一致

::::: steps

1. `mountElement` 函数的基本设计

   在 `render` 里已经区分出了首次渲染和更新，在首次渲染时，要做的事情其实就是把当前的 `vnode` 挂载到容器里。这一步的挂载对应的就是 `mountElement` 函数

   该函数接收一个元素类型的 `vnode`，然后把它变成真实 DOM，再插入到容器中

   :::info 实现思路

   子节点的处理，需要区分两种情况
  
   - 如果子节点是字符串，那么它就是一个文本节点，直接设置文本内容即可
   - 如果是一个数组，那么它就是多个子节点，则需要遍历子节点。对每个子节点进行 `patch` 即可

   :::

   :::code-tabs

   @tab renderer.ts

   ```ts
   function createRenderer(options) {
     // ...

     function mountElement(vnode, container) {
       const el = createElement(vnode.type);

       // 如果子节点是字符串，则表明他是文本节点
       if (typeof vnode.children === 'string') {
         setElementText(el, vnode.children);
       }

       else if (Array.isArray(vnode.children)) {
         vnode.children.forEach((child) => {
           patch(null, child, el);
         })
       }

       insert(el, container);
     }
   }

   const vnode1 = {
     type: "h1",
     children: "hello world",
   };

   const vnode2 = {
     type: "h1",
     children: [
       {
         type: 'p',
         children: 'Hello World',
       }
     ],
   };
   ```

   :::

2. 挂载元素属性

   子节点挂载以后，下一步就该处理元素本身的属性了。这里可以先给 `vnode` 增加一个 `props` 字段，用来描述元素上的属性

   ```ts
   const vnode = {
     type: "h1",
     props: {
       id: "foo",
     },
     children: [
       {
         type: "p",
         children: "Hello World",
       },
     ],
   };
   ```

   :::note 这个地方就存在非常多的细节问题

   因为 HTML attributes 和 DOM properties 并不是完全一一对应：

   - 有的名称不同，如 `class` 对应 `className`
   - 有的只存在于 HTML 属性中，如 `aria-*`
   - 有的只存在于 DOM properties 中，如 `textContent`
   - 在表单元素中 HTML 属性和 DOM properties 还可能出现 "值不同步" 的情况[+值不同]

   :::

   还有就是布尔属性的特殊情况，如 `disabled`

   ```ts
   // 这三种写法都会让按钮处于禁用状态
   el.setAttribute('disabled', '')
   el.setAttribute('disabled', 'false')
   el.setAttribute('disabled', false)


   // 而使用 DOM Properties 的方式时，只有第一种写法会让按钮处于禁用状态
   el.disabled = true
   el.disabled = false
   el.disabled = ''
   ```

   这些特殊情况虽然复杂，但把握住一个核心原则即可：==如果某个属性在元素对象上存在，就优先使用 DOM Properties 的方式去设置；否则才使用 setAttribute==。这个原则的好处是能够最大程度地保证虚拟节点里的值和真实 DOM 的行为保持一致

   :::info 实现思路
   只需补充一个判断，如果某个 key 本身就存在于元素对象上，那么优先按 `DOM Properties` 去处理；否则再退回 `setAttribute`
   :::

   :::code-tabs

   @tab renderer.ts

   ```ts
   function mountElement(vnode, container) {
     const el = createElement(vnode.type);

     // ...

     if (vnode.props) {
       for (const key in vnode.props) {
         const value = vnode.props[key];

         // 如果 key 在元素对象上存在，则优先使用 DOM Properties 的方式去设置
         if (key in el) {
           const type = typeof el[key];

           // 对于布尔属性，如果值是空字符串，也应该把它设置为 true 
           if (type === "boolean" && value === "") {
             el[key] = true;
           } else {
             el[key] = value;
           }
         } else {
           el.setAttribute(key, value);
         }
       }
     }

     insert(el, container);
   }
   ```

   :::

3. 只读 DOM 属性的处理

   即使加上了 `key in el` 这层判断，也还是会遇到例外情况。例如 `form` 属性，它虽然在元素对象上存在，但它是一个只读属性，无法通过 HTML properties 的方式去设置它的值

   ```html
   <form id="form1"></form>
   <input form="form1" />
   ```

   :::info 实现思路
   把 "这个属性到底该不该作为 DOM Properties 设置" 单独封装成一个函数，在函数中额外去判断这种特殊情况
   :::

   :::code-tabs

   @tab shouldSetAsProps.ts

   ```ts
   function shouldSetAsProps(el, key, value) {
     if (key === "form" && el.tagName === "INPUT") {
       return false;
     }

     return key in el;
   }
   ```

   @tab renderer.ts

   ```ts
   function shouldSetAsProps(el, key, value) {
     if (key === "form" && el.tagName === "INPUT") {
       return false;
     }

     return key in el;
   }

   function mountElement(vnode, container) {
     const el = createElement(vnode.type);

     // ...

     if (vnode.props) {
       for (const key in vnode.props) {
         const value = vnode.props[key];

         if (shouldSetAsProps(el, key, value)) { // [!code highlight]
           const type = typeof el[key];
           if (type === "boolean" && value === "") {
             el[key] = true;
           } else {
             el[key] = value;
           }
         } else {
           el.setAttribute(key, value);
         }
       }
     }

     insert(el, container);
   }
   ```

   :::

4. class 处理

   class 也属于 props 的一种特殊情况。在实际的 Vue 中，class 可以有三种写法：字符串、数组和对象

   :::code-tabs
   @tab 实际渲染

   ```html
   <button class="btn btn-primary"></button>
   <button :class={ btn: true, 'btn-primary': true }></button>
   <button :class="[ 'btn', 'btn-primary' ]"></button>
   ```

   @tab vnode

   ```ts
   const vnode1 = {
     type: "button",
     props: {
       class: "btn btn-primary",
     },
   }

   const vnode2 = {
     type: "button",
     props: {
       class: {
         btn: true,
         'btn-primary': true,
       },
     },
   }

   const vnode3 = {
     type: "button",
     props: {
       class: ['btn', 'btn-primary'],
     },
   }
   ```

   :::

   因此，对于这些情况，有必要在处理 class 的时候进行数据结构的转换，最终全部转换为字符串形式即可

   :::info 实现思路
   - 对于字符串类型，直接使用即可
   - 对于数组类型，先把数组中的每一项都转换成字符串，然后再用空格连接成一个新字符串
   - 对于对象类型，直接通过 `for...in` 循环，把对象中存在的属性拼接即可
   :::

   ```ts
   import { isString, isArray, isObject } from "@vue/shared";

   function normalizeClass(value) {
     let res = '';

     if (isString(value)) {
       res = value;
     }
     
     // 数组类型的判断要在对象类型的判断之前，因为数组也是对象
     else if (isArray(value)) {
       for (let i = 0; i < value.length; i++) {
         const normalized = normalizeClass(value[i]);
         if (normalized) {
           res += normalized + ' ';
         }
       }
     }

     else if (isObject(value)) {
       for (const name in value) {
         if (value[name]) {
           res += name + ' ';
         }
       }
     }

     return res.trim();
   }
   ```

5. 平台无关性处理

   不管是 class 还是其他的 HTML 属性是依赖于 DOM 的，因此它们的处理也不应该在 `runtime-core` 中进行，而应该放在 `runtime-dom` 中进行处理

   将上述逻辑进行抽离，放在 `runtime-dom` 中的 `patchProps` 函数中进行处理即可

   :::code-tabs

   @tab patchProps.ts

   ```ts
   function shouldSetAsProps(el, key, value) {
     if (key === 'form' && el.tagName === 'INPUT') {
       return false;
     }

     return key in el;
   } 

   const options = {
     createElement(tag) {
       return document.createElement(tag);
     },
     setElementText(el, text) {
       el.textContent = text;
     },
     insert(el, parent, anchor = null) {
       parent.insertBefore(el, anchor);
     },
     patchProps(el, key, prevValue, nextValue) {
       if (shouldSetAsProps(el, key, nextValue)) {
         const type = typeof el[key];
         if (type === "boolean" && nextValue === "") {
           el[key] = true;
         } else {
           el[key] = nextValue;
         }
       } else {
         el.setAttribute(key, nextValue);
       }
     }
   }
   ```

   @tab mountElement.ts

   ```ts
   function createRenderer(options) {
     const { createElement, insert, setElementText, patchProps } = options;

     function mountElement(vnode, container) {
       const el = createElement(vnode.type);

       if (typeof vnode.children === "string") {
         setElementText(el, vnode.children);
       } else if (Array.isArray(vnode.children)) {
         vnode.children.forEach((child) => {
           patch(null, child, el);
         });
       }

       if (vnode.props) {
         for (const key in vnode.props) {
           const value = vnode.props[key];
           patchProps(el, key, null, value);
         }
       }

       insert(el, container);
     }

     // ...
   }
   ```

   :::

   > [!IMPORTANT]
   > 在经过统计后，实际 `el.className` 设置样式的效率是最高的，因此在 `patchProps` 中对于 class 的处理，最终还是要通过 `el.className` 来设置的，也就无需再 `shouldSetAsProps` 中单独去判断 class 了

## 卸载的基本实现

::::: steps

1. 建立 `vnode` 和真实 DOM 之间的联系

   在 `render` 的实现中，若新 `vnode` 不存在，但旧 `vnode` 存在时。是通过 `container.innerHTML` 为空来清空容器的，这种做法虽然简单，但它只是把结果 "看起来删掉了"，并没有真正解决卸载过程中的边界问题

   :::note 真正的问题是

   - 组件的卸载生命周期不会执行
   - 自定义指令的卸载逻辑不会执行
   - 已绑定的事件、引用关系也都没有被正确清理
   :::

   > [!IMPORTANT]
   > 卸载的目标不是清空容器 ，而是根据 `vnode` 找到它对应的真实 DOM，再把这个 DOM 从父节点中移除

   :::info 实现思路
   建立 `vnode` 和真实 DOM 之间的联系
   :::

   ```ts
   function mountElement(vnode, container) {
     const el = vnode.el = createElement(vnode.type);

     // ..
   }
   ```

   这里在创建真实 DOM 元素时，会把真实 DOM 元素赋值给 `vnode.el` 属性，这样也就建立了虚拟节点和真实 DOM 之间的联系。一旦 `vnode.el` 建立起来，后续无论是更新还是卸载，都可以通过虚拟节点直接拿到真实 DOM

   ```ts
   function unmount(vnode) {
     const parent = vnode.el.parentNode;

     if (parent) {
       parent.removeChild(vnode.el);
     }
   }

   function render(vnode, container) {
     if (vnode) {
       patch(container._vnode, vnode, container)
     } else {
       if (container._vnode) {
         unmount(container._vnode);
       }
     }

     container._vnode = vnode;
   }
   ```

   这里先简单处理了卸载的逻辑，后续在组件和指令章节会补充更多细节

2. 不同类型节点的更新处理

   当新旧 `vnode` 的 `type` 不同时，说明它们之间没有任何可比性了，这时候就不应该再走更新的流程了，而是应该直接卸载旧节点，再挂载新节点

   例如第一次渲染的是 `p`，第二次渲染的是 `input`：

   ```ts
   const vnode1 = {
     type: "p",
   };

   const vnode2 = {
     type: "input",
   };
   ```
  
   这两种节点之间其实并不存在 "打补丁" 的意义。因为它们本身就是两种完全不同的元素类型，对应的属性、行为、子节点处理方式都可能不同

   :::info 实现思路
   1. 先卸载旧节点
   2. 挂载新节点
   :::

   ```ts
   function patch(oldVNode, newVNode, container) {
     // 如果新旧 vnode 的类型不同，则直接将旧 vnode 卸载
     if (oldVNode && oldVNode.type !== newVNode.type) {
       unmount(oldVNode);
       oldVNode = null; // 卸载后将 oldVNode 置空，以便后续逻辑按首次挂载处理
     }

     if (!oldVNode) {
       mountElement(newVNode, container);
     } else {
       // 更新逻辑
     }
   }
   ```

   到目前为止，默认 `vnode.type` 是字符串，也就是普通元素标签，例如 `div`、`p`、`input`。但在 Vue 的完整渲染体系中，`type` 还可能表示组件、`Fragment` 或其他内置类型

   因此，`patch` 最终不能只处理字符串类型，而是需要先根据 `type` 做分发处理

   ```ts
   function patch(oldVNode, newVNode, container) {
     if (oldVNode && oldVNode.type !== newVNode.type) {
       unmount(oldVNode);
       oldVNode = null;
     }

     const { type } = newVNode;

     if (typeof type === "string") {
       if (!oldVNode) {
         mountElement(newVNode, container);
       } else {
         // 元素更新
         console.log("打补丁");
       }
     } else if (typeof type === "object") {
       // 组件处理
       console.log("组件处理");
     } else {
       // 其他类型
       console.log("未知类型");
     }
   }
   ```

## 事件处理
