[+依赖收集]: 这个过程也被称为 "依赖收集"。通过数据，找到依赖它的函数
[+依赖追踪]: 这个过程也被称为 "依赖追踪"。通过函数执行过程，反向记录它用到了哪些数据
[+副作用函数]: 副作用函数在执行过程中，==会影响外部环境，或者依赖外部状态=={.important}。比如 `console.log`、DOM 操作、网络请求、读取全局变量等

Vue 的响应式，本质上是在 ==**函数** 和 **它运行时读取过的数据** 之间建立联系=={.important}。如果一个函数在执行时用到了某些数据，那么这些数据变化后，这个函数往往也应该重新执行

:::details 为什么不是所有函数都自动参与响应式
Vue 并不会强制所有函数都和数据建立关系，而是只处理那些需要响应更新的函数。需要依赖数据的函数，就建立关联；不需要依赖数据的函数，就不建立关联。这样做的好处是：

- 避免无意义的追踪
- 只在真正需要更新的地方触发重新执行
- 把控制权交给开发者，而不是把所有逻辑都塞进响应式系统
:::

## 核心流程

核心是让函数如何知晓依赖了哪些数据，因此实现响应式的关键点是：==数据要能被拦截，函数执行时要能被记录=={.important}。Vue 会先把数据做成响应式对象，在函数运行期间，记录它访问了哪些属性，最后当属性变化时，反向找到相关函数并重新执行

:::table full-width

| 阶段 | Vue 在做什么 | 目的 |
| --- | --- | --- |
| 1. 标记数据 | 把普通数据变成响应式数据 | 让读取和修改都可被感知 |
| 2. 执行函数 | 运行副作用函数[+副作用函数] | 收集它实际访问了哪些数据 |
| 3. 建立关系 | 把函数与它读取过的数据关联起来[+依赖收集] | 形成依赖关系 |
| 4. 触发更新 | 数据发生变化时，找到相关函数重新执行[+依赖追踪] | 完成自动更新 |

:::

换个角度看，就是这条链路：`函数执行` -> `读取数据` -> `建立依赖` -> `数据变化` -> `重新执行函数`

::::details 实现差异

:::table full-width

| 版本 | 实现方式 | 说明 |
| --- | --- | --- |
| Vue2 | `Object.defineProperty` | 通过属性劫持追踪读写 |
| Vue3 | `Proxy` | 通过代理对象统一拦截，更灵活，也更完整 |

:::

::::

## 实现流程

### 基本实现

::::steps

1. 先准备三个最基本的角色

   原始数据对象 `obj`、一个副作用函数 `effect`、一个依赖容器 `buckets`

   :::details 代码

   ```js
   const obj = {
     name: 'John',
     age: 20,
   };

   const buckets = new Set();

   function effect() {
     app.innerHTML = proxy.name;
   }
   ```

   :::

   这里的 `effect` 就是 "数据变化后需要重新执行的函数"，`buckets` 则用来存放这些函数

2. 用 `Proxy` 包一层，让读取和修改都能被拦截

   只有拦截住这两个动作，后面才有机会做依赖收集和触发更新

   :::details 代码

   ```js
   const proxy = new Proxy(obj, {
     get(target, key, receiver) {},
     set(target, key, value, receiver) {},
   });
   ```

   :::

3. 在 `get` 中做最简单的依赖收集

   当 `effect` 执行到 `proxy.name` 时，会触发 `get`。这时就把 `effect` 放进 `buckets`，表示这个函数依赖了当前读取的数据

   :::details 代码

   ```js
   get: (target, key, receiver) => {
     buckets.add(effect);
     return Reflect.get(target, key, receiver);
   }
   ```

   :::

4. 手动先执行一次 `effect`，完成首次渲染和首次收集

   这一步非常关键，因为如果 `effect` 从来没执行过，就不会触发 `get`，也就不会建立依赖关系

   :::details 代码

   ```js
   effect();
   ```

   :::

5. 在 `set` 中做最简单的触发更新

   当执行 `proxy.name = 'Jane'` 时，会触发 `set`。`set` 先把值真正写回原对象，遍历 `buckets`，把里面收集到的函数重新执行一遍

   这也就完成了：`数据变化` -> `找到相关函数` -> `重新执行函数` -> `页面重新渲染`

   :::details 代码

   ```js
   set: (target, key, value, receiver) => {
     const result = Reflect.set(target, key, value, receiver);
     buckets.forEach((fn) => fn());
     return result;
   }
   ```

   :::

6. 整条链路

   :::table full-width

   | 阶段 | 触发点 | 发生的事 |
   | --- | --- | --- |
   | 首次执行 | `effect()` | 触发副作用函数，开始读取响应式数据 |
   | 依赖收集 | `get` | 把 `effect` 收进 `buckets` |
   | 修改数据 | `proxy.name = 'Jane'` | 触发 `set` |
   | 派发更新 | `buckets.forEach(fn => fn())` | 把收集到的副作用函数重新执行 |
   | 更新视图 | `app.innerHTML = proxy.name` | DOM 显示最新值 |

   :::

### 细节处理

:::::steps

1. 处理 `target`

   由于 `Proxy` 的目标值必须是对象，如果传入的是基本类型，就没有办法继续做代理

   :::tip 这里的处理可以拆成两层来看：

   - 运行时：先用 `isObject(target)` 判断，如果不是对象就直接返回
   - 类型层：通过 `val is Record<any, any>` 这个自定义守卫，让 TypeScript 知道后面的 `target` 已经可以安全交给 `Proxy`
   :::

   ::::details 代码

   :::code-tabs

   @tab reactive.ts

   ```ts
   import { track, trigger } from './effect';
   import { isObject } from './utils';

   export function reactive<T extends object>(target: T): T;
   export function reactive<T>(target: T): T;

   export function reactive(target: unknown) {
     // reactive 只能代理对象，基本类型直接返回  // [!code focus]
     if (!isObject(target)) {  // [!code focus]
       return target; // [!code focus]
     }   // [!code focus]

     return new Proxy(target, {
       get: (target, key, receiver) => {
         track(target, key);
         return Reflect.get(target, key, receiver);
       },
       set: (target, key, value, receiver) => {
         const result = Reflect.set(target, key, value, receiver);

         trigger(target, key);
         return result;
       },
     });
   }
   ```

   @tab utils.ts

   ```ts
   export const isObject = (val: unknown): val is Record<any, any> => {
     return typeof val === 'object' && val !== null;
   };
   ```

   :::
   ::::

2. 处理同一个原始对象被重复代理

   如果同一个原始对象被重复调用 `reactive`，或者把一个已经代理过的对象再次传给 `reactive`，就可能创建出多个不同的代理对象

   :::tip 这里的处理也可以拆成两层来看：

   - 缓存层：用 `WeakMap` 缓存 "原始对象 -> 代理对象" 的映射如果同一个原始对象已经代理过，就直接返回之前的代理对象
   - 标记层：通过额外的标记位判断传进来的值是不是已经是代理对象，如果已经是代理对象，就直接返回它自己

   这里用 `WeakMap` 而不用 `Map`，是因为它的 key 只能是对象，并且当原始对象没有被其他地方引用时，可以被垃圾回收，更适合拿来做代理缓存
   :::

   ::::details 代码

   ```ts
   import { track, trigger } from './effect';
   import { isObject } from './utils';

   export const enum ReactiveFlags {
     IS_REACTIVE = '__v_isReactive',  // [!code focus]
     IS_READONLY = '__v_isReadonly',
     IS_SHALLOW = '__v_isShallow',
     RAW = '__v_raw',
     SKIP = '__v_skip',
   }

   export interface Target {
     [ReactiveFlags.IS_REACTIVE]?: boolean  // [!code focus]
     [ReactiveFlags.IS_READONLY]?: boolean
     [ReactiveFlags.IS_SHALLOW]?: boolean
     [ReactiveFlags.RAW]?: any
     [ReactiveFlags.SKIP]?: boolean
   }

   export const targetMap = new WeakMap<Target, any>()

   export function reactive<T extends object>(target: T): T
   export function reactive<T>(target: T): T

   export function reactive(target: unknown) {
     if (!isObject(target)) {
       return target
     }

     // 同一个原始对象重复代理，直接返回缓存结果  // [!code focus]
     if (targetMap.has(target)) { // [!code focus]
       return targetMap.get(target)  // [!code focus]
     }  // [!code focus]

     // 传进来的已经是代理对象，就直接返回  // [!code focus]
     if (target[ReactiveFlags.IS_REACTIVE]) { // [!code focus]
       return target // [!code focus]
     }  // [!code focus]

     const proxy = new Proxy(target, {
       get: (target, key, receiver) => {
         if (key === ReactiveFlags.IS_REACTIVE) { // [!code focus]
           return true // [!code focus]
         }  // [!code focus]

         track(target, key)
         return Reflect.get(target, key, receiver)
       },
       set: (target, key, value, receiver) => {
         const result = Reflect.set(target, key, value, receiver)
         trigger(target, key)
         return result
       },
     })

     targetMap.set(target, proxy)

     return proxy
   }
   ```

   ::::

3. 处理 this 指向问题

   如果对象内部存在 `getter / setter`，且里面用到了 `this`，那么这里的 `this` 应该指向代理对象，而不是原始对象。否则内部访问到的属性，就可能绕过代理，拿不到正确的依赖收集和触发更新能力

   :::tip 这里的核心点是：

   需要使用 `Proxy` 中 `get` 和 `set` 的最后一个参数 `receiver`，保证 `this` 指向代理对象

   :::

   ::::details 代码

   :::code-tabs
   @tab reactive.ts

   ```ts
   const proxy = new Proxy(target, {
     get: (target, key, receiver) => {
       if (key === ReactiveFlags.IS_REACTIVE) {
         return true;
       }

       track(target, key);

       // 把 receiver 传进去，保证 getter / 原型链场景中的 this 指向代理对象  // [!code focus]
       return Reflect.get(target, key, receiver);  // [!code focus]
     },
     set: (target, key, value, receiver) => {
       // 同理把 receiver 传给 Reflect.set，保证 setter 内部 this 正确  // [!code focus]
       const result = Reflect.set(target, key, value, receiver);  // [!code focus]

       trigger(target, key);
       return result;
     },
   });
   ```
