> [!NOTE]
> 词元（Token）是大模型里最基本的概念。它既是模型的 **输出单位**，也是模型的 **输入单位**——你发给模型的提示词（prompt），首先就会被拆成词元。

## NLP 发展的四个阶段

在讲词元之前，先把它放进整个 NLP 的发展脉络里：

::::steps

1. 起源
2. 基于规则
3. 基于统计 —— N-Gram、词袋模型
4. 深度学习与大数据驱动 —— 大语言模型

::::

词元就是从第 4 阶段开始，大模型处理文本的统一入口。

## 什么是词元

**每一个词元都有一个唯一的 ID**，所以最终真正喂给大模型的，是一个 **词元 ID 列表**。

举个例子，看一段英文怎么变成 ID 列表：

::::steps

1. 用户输入文本：`Learn about language model tokenization.`
2. 先分词：`[Learn] [about] [language] [model] [token] [ization] [.]`
3. 查词表映射 ID：每个 token 查表换成一个整数，如 `[1122, 98, 4012, ...]`
4. 最终进入模型：`[1122, 98, 4012, 3305, 2351, 7489, 13]` 这样的整数列表

::::

> [!TIP]
> OpenAI 提供了在线查看词元的工具：<https://platform.openai.com/tokenizer>，可以直观看到一段文本被切成了哪些 token。

## 四种分词策略

### 1. 词级分词

直接以"完整的词"为单位切分，适用于空格分隔的语言（如英语）。

```text
I love natural language processing.
```

```text
["I", "love", "natural", "language", "processing", "."]
```

特点：粒度大、语义清晰，对英文常见词效果好。**缺点是无法处理未登录词**——遇到新词、自造词就不认识了，即不支持 OOV。

> [!NOTE]
> OOV 全称 Out Of Vocabulary，表示词超出了词汇表，模型无法处理。

### 2. 子词级分词

把词切成更小的子词单元（词干、前缀、后缀），适合处理未知词。

```text
lovely  →  love、ly
```

特点：兼顾词级和字符级的优点，**支持 OOV**，能处理新词和自造词。

### 3. 字符级分词

把每个字符当作一个词元，不依赖词典。

```text
Hello  →  ["H", "e", "l", "l", "o"]
```

特点：适合拼写敏感任务（语言模型、自动补全）；能处理所有字符，**OOV 问题彻底消失**（不存在造不出来的词）。缺点是序列变长，难以建模高级语义。

### 4. 字节级分词

先把文本按 UTF-8 **字节** 切分，再对字节序列建模。基本单位是字节而非字符或词，因此 **具有语言无关性**。

```text
Hello 😊
```

```text
[72, 101, 108, 108, 111, 32, 240, 159, 152, 138]
```

- 前 6 个是 ASCII 字符（H e l l o 空格）
- 后 4 个是 emoji 的 UTF-8 编码

优势：

1. 语言无关，对多语言友好
2. 处理 OOV 能力强
3. 压缩效率高：根据训练数据频率自动构建最佳子词组合，减少 token 总数

## 不同模型的分词差异

目前主流大模型（GPT 系列、DeepSeek、Kimi 等）基本都采用 **字节级 BPE 分词**。但不同版本、不同模型的切分结果会不一样。

同一模型不同版本会进化。例如 Python 的 `elif`，在 GPT-2 里被切成两个词 `[el、if]`，到 GPT-4 已经有了自己的词元 `[elif]`——这源于模型对代码关注度的提升。词表规模也在变大：GPT-4 用 `cl100k_base`（约 10 万词表），到 GPT-4o / GPT-5 升级为 `o200k_base`（约 20 万，GPT-5 另有 `o200k_harmony` 变体）。**词表越大，同样的文本切出的 token 越少**，既省上下文也对多语言更友好。

不同模型差异更明显，以 `ChatGPT真厉害！` 为例：

:::tabs

@tab GPT-4o / GPT-5

```text
['Chat', 'GPT', '真', '厉害', '！']
```

@tab GPT-2 / GPT-4

```text
['Chat', 'G', 'PT', '真', '厉', '害', '！']
```

@tab BERT

```text
['[UNK]', '真', '厉', '害', '！']
```

@tab Phi-3

```text
['▁Chat', 'G', 'PT', '真', '厉', '害', '！']
```

:::

各主流模型的分词方法对比：

:::table full-width

| 模型 | 分词方法 | 词表规模 | 字节级 | OOV 问题 |
| --- | --- | --- | --- | --- |
| GPT-2 | Byte-level BPE（`tiktoken`） | ~5 万 | 是 | 无 |
| GPT-4 | Byte-level BPE（`cl100k_base`） | ~10 万 | 是 | 无 |
| GPT-4o / GPT-5 | Byte-level BPE（`o200k_base`） | ~20 万 | 是 | 无 |
| DeepSeek-V3 | Byte-level BPE | ~12.8 万 | 是 | 无 |
| Kimi K2 | Byte-level BPE | 大词表 | 是 | 无 |
| BERT | WordPiece | ~3 万 | 否 | 有 `[UNK]` |

:::

> [!NOTE]
> **Claude 的分词器并不公开**。Anthropic 没有像 OpenAI 的 `tiktoken` 那样开源分词器，只提供一个 token 计数 API。所以想精确知道 Claude 怎么切词、词表多大，目前是做不到的——这也是闭源模型的常态。

## 小结

词元是大模型读写文本的最小单位，文本进出模型都要经过"分词 → 查表换 ID"这一步。从词级到字节级，分词策略的演进核心就是 **如何更好地处理未登录词（OOV）与多语言**；而现代主流模型（GPT-5、DeepSeek、Kimi 等）几乎都落在 **字节级 BPE** 上，只是词表规模和具体合并规则不同。理解了词元，才能理解后面 Embedding 是怎么把这些 ID 变成有语义的向量的。
