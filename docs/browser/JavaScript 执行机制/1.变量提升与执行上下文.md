很多人第一次接触 `hoisting` 时，会把它理解为“代码被移动到了最前面”。
这个说法便于入门，但不够准确。

更准确的表述是：==声明不会在源码里移动，而是在编译阶段被记录到执行上下文的环境结构中==。

## 先看一个经典例子

```js
showName()
console.log(myName)

var myName = '极客时间'
function showName() {
  console.log('showName 函数被执行')
}
```

输出：

```txt
showName 函数被执行
undefined
```

:::details 为什么是 `undefined` 而不是报错
因为 `var myName` 的“声明”在编译阶段已经进入环境记录，并被初始化为 `undefined`。执行阶段遇到 `console.log(myName)` 时可以找到这个变量，只是还没执行赋值语句。
:::

## JavaScript 执行不是“一步到位”

一段代码通常经历这条主线：

1. 源代码输入。
2. 编译阶段（构建执行上下文与可执行表示）。
3. 执行阶段（按顺序执行）。

![JS 的执行流程图](https://cdn.nlark.com/yuque/0/2023/png/28960762/1700723894183-8a2be3df-f011-4be3-ba15-0162c7f7d438.png)

## 编译阶段到底做了什么

::::steps
1. 创建执行上下文（Execution Context）

   全局代码会创建全局执行上下文，函数调用时会创建函数执行上下文。

2. 构建环境记录

   可以先用“变量环境（VE）+ 词法环境（LE）”的模型来理解：

   - `var` 与函数声明主要进入变量环境
   - `let/const` 与块级作用域绑定更偏向词法环境

3. 收集声明

   - `var`：创建同名条目并初始化为 `undefined`
   - 函数声明：创建同名条目并直接绑定函数体

4. 生成可执行代码，进入执行阶段
::::

可以用一个抽象结构辅助理解：

```js
VariableEnvironment = {
  myName: undefined,
  showName: function showName() {},
}
```

![编译阶段和环境构建示意](https://cdn.nlark.com/yuque/0/2024/png/28960762/1704253350765-a46627a7-a924-4caa-b0fb-7f3e41c36b21.png)

## 执行阶段怎么运行

执行阶段仍然是“自上而下逐行执行”，只是变量查找会先从当前执行上下文的环境里取值。

::::steps
1. 执行 `showName()`：在当前环境找到函数引用并调用。
2. 执行 `console.log(myName)`：在环境中命中 `myName`，值为 `undefined`。
3. 执行 `myName = '极客时间'`：把同名条目的值从 `undefined` 更新为字符串。
::::

## 执行上下文有哪些类型

:::table full-width
| 类型 | 触发时机 | 生命周期 |
| --- | --- | --- |
| 全局执行上下文 | 页面脚本开始执行 | 与页面生命周期一致（通常只有一份） |
| 函数执行上下文 | 每次函数调用 | 调用结束后可被销毁 |
| `eval` 执行上下文 | 执行 `eval` 字符串代码 | 取决于 `eval` 调用时机 |
:::

## 同名声明如何处理

看这段代码：

```js
function showName() {
  console.log('极客邦')
}
showName()

function showName() {
  console.log('极客时间')
}
showName()
```

结果两次都会输出 `极客时间`。

原因：

1. 编译阶段收集函数声明时，同名项后者会覆盖前者。
2. 执行阶段查找 `showName` 时，只能拿到最终保留的那份绑定。

## 小结

1. 变量提升不是“移动代码”，是“编译阶段提前建立绑定”。
2. 执行上下文是代码运行时的环境抽象，桥接了编译和执行。
3. 弄清执行上下文后，很多“看起来反直觉”的现象都会变得可解释。
