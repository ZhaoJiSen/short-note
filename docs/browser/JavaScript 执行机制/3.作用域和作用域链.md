作用域（Scope）决定了变量和函数“在哪里可访问”，以及“什么时候会失效”。
作用域链（Scope Chain）决定了变量查找时“按什么路径向外层查找”。

## 先看作用域

:::table full-width
| 作用域类型 | 核心特点 | 生命周期 |
| --- | --- | --- |
| 全局作用域 | 全局可访问（受模块边界影响） | 通常与页面生命周期一致 |
| 函数作用域 | 仅函数内部可访问 | 函数执行相关 |
| 块级作用域（ES6+） | `let/const` 受 `{}` 限制 | 块执行期间 |
:::

## 一个常见疑问：变量到底取哪一个

```js
function bar() {
  console.log(myName)
}

function foo() {
  var myName = '极客邦'
  bar()
}

var myName = '极客时间'
foo()
```

`bar` 最终输出的是 `极客时间`，不是 `极客邦`。

这正是作用域链和词法作用域一起决定的结果。

## 作用域链如何查找变量

每个执行上下文除了自己的环境记录外，还会持有一个指向外层环境的引用（可理解为 `outer`）。
变量查找规则是：

::::steps
1. 先查当前执行上下文。
2. 当前找不到，就沿着 `outer` 去外层查。
3. 一直查到全局。
4. 最后仍找不到，抛出 `ReferenceError`。
::::

![带有 outer 引用的调用栈示意图](https://cdn.nlark.com/yuque/0/2023/png/28960762/1701239111852-739170d5-6659-4da8-ad05-f0a6098df004.png)

## 词法作用域：为什么跟“调用位置”无关

词法作用域是静态的，在“代码定义时”就确定，而不是执行时临时决定。

这解释了前面的例子：

1. `bar` 定义在全局作用域，它的外层链路指向全局。
2. 即使 `bar()` 在 `foo` 里被调用，也不会改写它的词法外层。
3. 所以 `bar` 读取 `myName` 时，会命中全局的 `myName`。

## 块级作用域里的变量查找

```js
function bar() {
  var myName = '极客世界'
  let test1 = 100

  if (1) {
    let myName = 'Chrome浏览器'
    console.log(test)
  }
}

function foo() {
  var myName = '极客邦'
  let test = 2

  {
    let test = 3
    bar()
  }
}

var myName = '极客时间'
let myAge = 10
let test = 1
foo()
```

这个例子展示了两件事：

1. `let/const` 会把变量绑定到对应块级词法环境。
2. 查找变量时依旧遵守“当前 -> 外层”的链式规则。

## 关于函数的 `[[Scope]]`

`[[Scope]]` 是函数内部的隐式属性（引擎内部槽位，不是业务代码可直接访问的普通属性）。

它可以帮助理解两件事：

1. 函数在创建时会关联其词法环境链路。
2. 函数执行时会基于这条链路做变量解析。

所以可以把它理解为：函数“记住了它定义时能访问哪些外层环境”。

## 小结

1. 作用域解决“变量在哪可见”。
2. 作用域链解决“变量找不到时去哪里继续找”。
3. 词法作用域是静态规则，这是理解闭包与 this/调用位置差异的关键前提。
