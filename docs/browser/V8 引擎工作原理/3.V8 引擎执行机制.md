V8 不是单纯的“解释执行”，也不是传统“只编译一次”的模式。
它采用的是解释器与 JIT 编译器协作的执行架构：先让代码快速跑起来，再对热点代码持续提速。

![V8 导图](/browser/source-images/1710905216772-b93173b7-e536-4fe4-a782-5b1d3358ad67.jpeg)

## 先看编译型与解释型差异

在任何语言里，源代码都不能直接被 CPU 执行，必须经过“翻译”。

:::table full-width

| 模式 | 典型流程 | 优势 | 代价 |
| --- | --- | --- | --- |
| 编译型 | 源码 -> 编译 -> 机器码文件 -> 执行 | 运行效率高 | 编译耗时、产物占空间 |
| 解释型 | 源码 -> 解释器逐步翻译执行 | 启动快、灵活 | 热路径性能较弱 |
:::

![编译器 "翻译" 代码](/browser/source-images/1700121686878-e59f7705-c7c1-4477-b139-c0c0cc27845e.png)

![解释器 "翻译" 代码](/browser/source-images/1700122292773-f0d4e8a8-37c4-4627-9310-f8943b58fede.png)

V8 的设计目标是同时拿到两者优点：

1. 启动阶段要快（解释器先上）。
2. 长时间运行要快（JIT 优化热点）。

## V8 执行主线：源码到机器码

![V8 执行流程](/browser/source-images/1710906476145-d8c16797-9588-447b-83bd-3806e3eb245a.png)

:::::steps

1. 词法分析（Tokenize）

   源代码先被拆成 Token（关键字、标识符、字面量、操作符等最小语法单元）。

   ![token 示意](/browser/source-images/1710907154660-74acf8f3-eb72-4feb-bdbe-4554e0393699.png)

2. 语法分析（Parse）

   Token 按语法规则组合成 AST（抽象语法树）。
   如果语法不合法，解析阶段就会抛错并中止。

   ![AST 结构](/browser/source-images/1710906859569-aced18f1-0e62-47e5-b871-bbd8f7abd5d4.png)

3. Ignition 根据 AST 生成字节码

   字节码是介于 AST 和机器码之间的中间表示，便于解释执行。

4. 解释执行字节码

   首次执行时，Ignition 会按字节码逐条执行，并收集运行时反馈信息。

5. 热点代码触发 TurboFan 优化

   某段代码执行频率高、类型信息稳定时，TurboFan 把它编译成机器码并缓存。

6. 后续直接走优化后的机器码

   同一热点路径再次命中时，执行速度明显提升。

:::::

## 为什么 V8 先生成字节码，不直接全量机器码

早期引擎方案曾更偏向“更早机器码”，但在复杂应用和移动端场景下，机器码体积与内存占用成本过高。
引入字节码后，启动与内存开销更可控，再配合热点优化拿到运行期性能。

![字节码与机器码占用空间对比](/browser/source-images/1710911907842-71f96574-069b-4799-8a95-b26aae1bfbc2.png)

## JIT：解释执行与编译优化如何协作

JIT（Just-In-Time）不是“抛弃解释器”，而是解释器和编译器并行协作。

![JIT 技术](/browser/source-images/1710912584039-c3f678d5-9bbe-4fd2-a3e7-27825bb1ec1d.png)

:::::steps

1. Ignition 先把字节码跑起来，保证启动时延。
2. 执行过程中持续采集类型反馈和调用次数。
3. 当函数或代码块变热，TurboFan 介入优化编译。
4. 将热点路径替换为机器码执行，提高吞吐。
5. 若运行时条件变化导致优化假设失效，会触发反优化（deopt）回退。

:::::

:::details Ignition 和 TurboFan 名字为什么这么起
`Ignition` 是“点火器”，表示先把程序点起来。
`TurboFan` 是“涡轮增压”，表示热点路径再提速。
这个命名本身就对应 V8 的执行哲学：先启动，再加速。
:::

## 用一段代码理解“热点优化 + 反优化”

```js
function add(a, b) {
  return a + b
}

add(1, 2)
add(3, 4)
add(5, 6)

// 若后续长期混入不同形态数据，可能触发优化路径回退
add('x', 1)
```

这段代码的关键点不是“能不能运行”，而是“类型形态是否稳定”。
热路径如果保持稳定，优化更容易命中；形态频繁变化，反优化概率就会上升。

## 这章在实战中的价值

:::table full-width

| 现象 | 底层原因 | 优化方向 |
| --- | --- | --- |
| 首次打开慢 | 解析/字节码生成成本高 | 降低首屏 JS 体积、按需加载 |
| 运行一段时间后变快 | 热点被 TurboFan 编译 | 关注热路径代码可预测性 |
| 性能忽高忽低 | 优化与反优化反复切换 | 保持数据结构与参数类型稳定 |
:::

理解 V8 执行机制后，你在做性能优化时就不会只盯“语法层写法”，而是能判断代码是否有利于引擎建立稳定优化路径。
