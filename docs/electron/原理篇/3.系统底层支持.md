## 应用入口文件的加载

每个 Electron 应用都是从主进程的入口脚本开始执行的。

之前我们提到了一个 `LoadEmbedderJavaScriptSource` 方法。当 Electron 执行这个方法时，实际上会执行一系列 TS 相关的代码，其中就有一个 TS 文件 `lib/browser/init.ts`。该文件实际上完成了加载应用入口脚本的工作。

[+searchPaths]: 在 `lib/browser/init.ts` 中，`searchPaths` 数组定义了 Electron 查找入口文件的顺序

在该文件中，有一个 `searchPaths` 数组。该数组中值的顺序，其实就是 Electron 查找对应入口文件的顺序。

:::: steps

1. 查找 app 子目录

   Electron 首先会在 `app` 子目录中查找入口文件

2. 查找 app.asar

   如果没有找到，继续查找 `app.asar` 文件

3. 查找 default_app.asar

   如果还是没有找到，会查找 `default_app.asar`

4. 报异常并退出

   如果以上都没有找到，那么就会报异常并退出
::::

:::warning 注意
如果命中了某一个文件，那么就会去读取对应的 `package.json` 文件
:::

读取 `package.json` 文件之后，就会根据里面的信息执行一系列的配置逻辑：

- 设置应用默认版本
- 设置应用默认的 `DesktopName`

配置工作完成之后，Electron 就会把 `package.json` 里面所对应的 `main` 的脚本交给 Node.js 来执行

:::tip 关于 mainStartupScript
`mainStartupScript` 对应的是主进程的入口脚本，如果 `package.json` 里面没有配置 `main` 属性，那么默认为 `index.js`
:::

另外，还有一个 `Module` 模块。这是 Node.js 提供的一个内置模块，这里会用到：

- `_resolveFilename`
- `_load`：就是负责加载并且执行开发者所提供的主进程所对应的入口脚本文件

## 提供系统底层支持

有关 Electron 自身提供的 API 能力，在前面我们介绍过，它们是由 `lib` 目录下面一系列的 TS 文件提供的。

但是仅仅依靠 TS 无法直接完成这些与操作系统相关的工作，这些工作的底层实现最终还是要由 C++ 代码来完成。

### 注册扩展模块

之前，我们在介绍 Electron 中初始化 Node.js 环境期间，会执行 `NodeBindings` 对象的 `Initialize` 方法，该方法内部，又会调用一个名为 `RegisterBuiltinModules` 的方法（`shell/common/node_bindings.cc`），该方法会注册 Electron 为 Node.js 提供的扩展库

[+ELECTRON_BUILTIN_MODULES]: 这个宏对应的代码，实际上就是在注册 Electron 扩展模块的底层支持模块

`ELECTRON_BUILTIN_MODULES` 这个宏对应的代码，实际上就是在注册 Electron 扩展模块的底层支持模块

```c++
// shell/common/node_bindings.cc
// 注册 Electron 为 Node.js 提供的扩展模块
void NodeBindings::RegisterBuiltinModules() {
  // ELECTRON_BUILTIN_MODULES 宏展开后会注册所有 Electron 扩展模块
  // 包括：clipboard、crash-reporter、dialog、shell 等
  ELECTRON_BUILTIN_MODULES
}
```

### app 模块的初始化

以 `app` 模块为例，里面也对应着一些宏。通过这些宏，最终能够定位到 `Initialize` 方法。

:::table full-width

| 组件 | 说明 |
| --- | --- |
| 字典的键 | `app` |
| 字典的值 | 一个异步对象 |
| 首次使用 | 执行异步对象的初始化方法 |
| 后续使用 | 从缓存中获取 |

:::

在该方法中有一个字典，字典对应的键就是 `app`，对应的值是一个异步对象。之后上层代码在首次使用这个 `app` 对象时，就会执行这个异步对象的初始化方法；之后再次使用时，都会从缓存中获取。

```c++
// shell/browser/api/electron_api_app.cc
// app 模块的初始化方法
void Initialize(v8::Isolate* isolate, v8::Local<v8::Object> exports) {
  // 创建 app 对象
  v8::Local<v8::Object> app = CreateApp(isolate);
  // 将 app 对象绑定到 exports
  exports->Set(context, gin::StringToSymbol(isolate, "app"), app);
}
```

## API 支持不同的操作系统

这里以 `AddRecentDocument` 方法为例，不同的操作系统，实现写在了不同的文件里面

:::table full-width

| 操作系统 | 实现文件 | 说明 |
| --- | --- | --- |
| Windows | `browser_win.cc` | Windows 系统实现 |
| Mac | `browser_mac.mm` | macOS 系统实现 |
| Linux | `browser_linux.cc` | 空方法，不支持 |

:::

这里我们就可以看出，同一个 API，为了支持不同的操作系统，Electron 中将实现写在了不同的文件里面

### Windows 实现

```c++
// shell/browser/browser_win.cc
// Windows 平台的 AddRecentDocument 实现
void Browser::AddRecentDocument(const base::FilePath& path) {
  // 使用 Windows API 添加到最近文档列表
  // SHAddToRecentDocs 会将文件添加到 Windows 的"最近使用的文件"列表
  SHAddToRecentDocs(SHARD_PATHW, path.value().c_str());
}
```

### macOS 实现

```objc
// shell/browser/browser_mac.mm
// macOS 平台的 AddRecentDocument 实现
void Browser::AddRecentDocument(const base::FilePath& path) {
  // 使用 macOS 的 NSDocumentController 添加到最近文档列表
  NSString* path_string = base::SysUTF8ToNSString(path.value());
  NSURL* url = [NSURL fileURLWithPath:path_string];
  [[NSDocumentController sharedDocumentController]
      noteNewRecentDocumentURL:url];
}
```

### Linux 实现

```c++
// shell/browser/browser_linux.cc
// Linux 平台的 AddRecentDocument 实现
// Linux 没有统一的最近文档标准，所以为空实现
void Browser::AddRecentDocument(const base::FilePath& path) {
  // 不支持
}
```

### 编译配置

接下来在 `filenames.gni` 文件中定义了编译文件的路径数组，在这个数组里面：

- 前面三项定义了不同操作系统下执行编译时需要参与编译的文件
- 数组的最后一项表示的是三个系统都需要编译的文件

之后，会在 `BUILD.gn` 文件里面使用这三个数组

:::details filenames.gni 文件结构示例

```gn
filenames = {
  # Windows 特有文件
  windows_sources = [
    "shell/browser/browser_win.cc",
    # ...
  ]

  # macOS 特有文件
  mac_sources = [
    "shell/browser/browser_mac.mm",
    # ...
  ]

  # Linux 特有文件
  linux_sources = [
    "shell/browser/browser_linux.cc",
    # ...
  ]

  # 所有系统共享的文件
  common_sources = [
    "shell/browser/browser.cc",
    # ...
  ]
}
```

:::

:::note
这种按操作系统分离实现的方式，使得 Electron 能够在不同平台上提供一致的 API 接口，同时还能针对每个平台进行特定的优化和适配。
:::
