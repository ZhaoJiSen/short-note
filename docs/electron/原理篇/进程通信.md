## Chromium 的 Mojo 框架

在 Chromium 内部，进程之间的通信是使用的一个名为 Mojo 的框架

Mojo 主要是提供了一套底层的 IPC 实现：

- 消息管道
- 数据管道
- 共享缓冲区
- ...

:::tip 理解 Mojo
Chromium 在 Mojo 的基础上，又做了一层封装：

- 简化不同语言（C++、Java、JS）之间的消息传递
- 简化不同进程之间的消息传递
:::

说回 Electron，因为 Electron 内部集成了 Chromium，因此 Electron 中进程间的通信也是通过 Mojo 来实现的

## 通信接口定义

在 `api.mojom` 文件中（源码地址：`shell/common/api/api.mojom`）定义了一系列的通信接口描述，其中的 `ElectronRenderer` 和 `ElectronBrowser` 这两个接口就和主进程以及渲染进程的通信有关了

:::table full-width

| 接口名称 | 所在进程 | 作用 |
| --- | --- | --- |
| `ElectronRenderer` | 渲染进程 | 定义渲染进程提供的能力 |
| `ElectronBrowser` | 主进程 | 定义主进程提供的能力 |

:::

之后，Electron 会在 `BUILD.gn` 文件中把 `api.mojom` 这个文件添加到编译配置文件中，接下来在编译 Electron 源码的时候，Mojo 框架会把这两个通信接口：

- 转译为具体的实现代码
- 将其写入到 `shell/common/api/api.mojom.h` 头文件里面

之后有两个文件：

- `shell/renderer/api/electron_api_ipc_renderer.cc`
- `shell/browser/api/electron_api_web_contents.cc`

就都会引用上面的头文件，也就是说，渲染进程的底层逻辑和主进程的底层逻辑都引用了上面的头文件

## invoke 通信的底层流程

当 Electron 开发者使用 `invoke` 来向主进程进行通信的时候，实际上执行的是位于 `shell/renderer/api/electron_api_ipc_renderer.cc` 里面的 C++ 代码，具体来讲是内部的一个名为 `Invoke` 的方法

:::: steps

1. 渲染进程创建 Promise

   在 `electron_api_ipc_renderer.cc` 文件内部的 `Invoke` 方法中，首先创建了一个 Promise

   ```c++
   // shell/renderer/api/electron_api_ipc_renderer.cc
   v8::Local<v8::Promise> Invoke(v8::Isolate* isolate,
                                  const std::string& channel,
                                  v8::Local<v8::Value> args) {
     // 创建 Promise，用于等待主进程返回结果
     v8::Promise::Resolver resolver =
         v8::Promise::Resolver::New(context).ToLocalChecked();
     // ...
   }
   ```

2. 调用 Mojo 通信对象

   之后调用了 `electron_browser_remote_` 这个对象的 `Invoke` 方法，而 `electron_browser_remote_` 这个对象正是 Mojo 的一个通信对象

   ```c++
   // shell/renderer/api/electron_api_ipc_renderer.cc
   // 通过 Mojo 通信对象向主进程发送消息
   electron_browser_remote_->Invoke(
       channel, std::move(args),
       base::BindOnce(&OnInvokeResponse, std::move(resolver)));
   ```

3. Mojo 发送消息

   当调用了 Mojo 通信对象的 `Invoke` 方法之后，Mojo 会组织要传递给主进程的数据和消息，然后发送给主进程

4. 主进程接收消息

   主进程接收到消息，最终会执行 `electron_api_web_contents.cc` 文件里面的 `WebContents::Invoke` 方法

   ```c++
   // shell/browser/api/electron_api_web_contents.cc
   // 主进程接收渲染进程的 invoke 消息
   void WebContents::Invoke(const std::string& channel,
                            base::Value::List args,
                            InvokeCallback callback) {
     // 触发 -ipc-invoke 事件
     EmitWithSender("-ipc-invoke", web_contents(), std::move(callback),
                    std::move(args));
   }
   ```

5. 触发事件处理

   在 `WebContents::Invoke` 方法内部，会发射一个名为 `-ipc-invoke` 的事件，并且将渲染进程传递过来的数据也一并发射过去，这个事件会触发位于 `lib/browser/api/web-contents.ts` 相关的处理逻辑
::::

## 事件处理逻辑

在 `web-contents.ts` 这个 TS 文件中，就监听了 `-ipc-invoke` 事件，在对应的具体的处理逻辑里面，会去查找一个 Map 对象

之所以要用到这个 Map 对象，是因为需要看用户是否在这个 Map 对象里面注册了自己的处理逻辑：

- 如果有，那么就执行用户提供的业务代码
- 如果没有，抛出异常

```typescript
// lib/browser/api/web-contents.ts
// 存储用户通过 ipcMain.handle 注册的处理逻辑
// key: 频道名，value: 处理函数
const invokeHandlers = new Map<string, Function>()

// 监听 -ipc-invoke 事件
webContents.on('-ipc-invoke', (event, channel, ...args) => {
  // 从 Map 中查找用户注册的处理函数
  const handler = invokeHandlers.get(channel)

  if (handler) {
    // 如果找到了，执行处理函数并返回结果
    handler(event, ...args)
  } else {
    // 如果没有找到，抛出异常
    throw new Error(`No handler registered for '${channel}'`)
  }
})
```

:::warning 注意
因为是 `invoke` 方法，所以还需要将处理结果返回给渲染进程，这个过程是由 `event.sendReply` 来实现的，具体对应的源码位置在 `shell/browser/api/event.cc`
:::

```c++
// shell/browser/api/event.cc
// sendReply 方法将处理结果返回给渲染进程
void Event::SendReply(v8::Local<v8::Value> result) {
  // 通过 Mojo 回调将结果发送回渲染进程
  // 渲染进程的 Promise 会被 resolve
  if (callback_) {
    std::move(callback_).Run(base::Value(result));
  }
}
```

## ipcMain.handle 的底层实现

开发者可以通过 `ipcMain.handle` 来为主进程注册某一个事件的处理逻辑，实际上底层对应的代码所做的事情，仅仅是把用户的处理逻辑包装起来存放到 Map 对象里面，到时候事件发生的时候就可以进行调用

```typescript
// lib/browser/api/ipc-main.ts
// ipcMain.handle 的实现
function handle(channel: string, handler: Function) {
  // 将用户的处理函数存入 Map 对象
  invokeHandlers.set(channel, handler)
}

// ipcMain.removeHandle 的实现
function removeHandle(channel: string) {
  // 从 Map 对象中移除处理函数
  invokeHandlers.delete(channel)
}
```

```text
ipcMain.handle('channel', handler)
        ↓
handler 存入 Map 对象
        ↓
渲染进程 invoke('channel')
        ↓
Mojo 传递消息到主进程
        ↓
主进程触发 -ipc-invoke 事件
        ↓
从 Map 中查找 handler
        ↓
执行 handler 并返回结果
```

> [!NOTE]
> 整个 invoke/handle 的通信流程，本质上是：渲染进程通过 Mojo 发送消息 -> 主进程接收并触发事件 -> 从 Map 中查找注册的处理逻辑 -> 执行并返回结果
