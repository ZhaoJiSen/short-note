Electron 之所以能同时拥有“桌面应用能力”和“网页渲染能力”，核心就在于它不是单进程模型，而是由==主进程==和一个或多个==渲染进程==共同组成。进程一旦拆开，通信就成了基础能力。

这章要解决的不是“API 怎么背”，而是另外三个更关键的问题：

- 为什么 Electron 要把能力拆到不同进程
- 主进程和渲染进程分别适合做什么
- 当多个窗口、多个页面同时存在时，消息应该如何安全地流动

:::tip 先建立一个判断标准
主进程更像“桌面操作系统代理”，负责窗口、菜单、系统 API、文件系统等能力；渲染进程更像“页面应用”，负责界面、交互和用户输入。
:::

## 主进程与渲染进程通信

主进程和渲染进程的通信，本质上是在解决一个边界问题：==页面想要调用桌面能力，但这些能力并不应该直接暴露给页面==。

Electron 为这件事提供了两组核心模块：

:::table full-width

| 模块 | 所在进程 | 作用 | 常见方法 |
| --- | --- | --- | --- |
| `ipcMain` | 主进程 | 监听渲染进程发来的消息 | `on`、`once`、`handle` |
| `ipcRenderer` | 渲染进程 | 向主进程发送消息、监听主进程消息 | `send`、`invoke`、`on` |

:::

这里可以把通信方式简单分成两类：

- `send/on`：更像“发通知”，适合单向触发
- `invoke/handle`：更像“发请求并等待结果”，适合打开对话框、读取文件、查询配置这类操作

### 为什么它能工作

课件里把 `ipcMain` / `ipcRenderer` 类比成了 `EventEmitter`，这个理解很重要。因为它能帮助你明白：所谓 IPC，并不神秘，本质上也是==频道 + 回调==的事件驱动模型。

:::code-tabs

@tab event.js

```js
const EventEmitter = require('events').EventEmitter
const event = new EventEmitter()

// 监听自定义事件
// 这里的角色很像 ipcMain.on / ipcRenderer.on
// 都是在某个频道上挂一个监听器
// 等待消息抵达后再执行逻辑
// eslint-disable-next-line no-console
event.on('some_event', () => {
  console.log('事件已触发')
})

module.exports = event
```

@tab index.js

```js
const event = require('./event')

// 主动触发某个频道的消息
// 这和 ipcRenderer.send('channel') 的思路是一致的
event.emit('some_event')
```

:::

这组示例不是在教你“用 EventEmitter 代替 IPC”，而是帮你先理解消息模型：

- 要先定义频道名
- 再绑定监听器
- 然后由另一侧触发该频道

### `send/on`：更适合通知型消息

当渲染进程只是想告诉主进程“某个动作发生了”，而不是等待一个返回值时，用 `send/on` 更合适。

比如：

- 用户点击了“新建窗口”按钮
- 用户触发了“保存草稿”动作
- 某个页面想通知主进程记录日志

:::code-tabs

@tab main.js

```js
const { app, BrowserWindow, ipcMain } = require('electron')

function createWindow() {
  const win = new BrowserWindow({
    width: 800,
    height: 600,
    webPreferences: {
      preload: require('node:path').join(__dirname, 'preload.js'),
    },
  })

  win.loadFile('index.html')
}

ipcMain.on('window:minimize', (event) => {
  const currentWindow = BrowserWindow.fromWebContents(event.sender)
  currentWindow?.minimize()
})

app.whenReady().then(createWindow)
```

@tab preload.js

```js
const { contextBridge, ipcRenderer } = require('electron')

contextBridge.exposeInMainWorld('electronAPI', {
  minimizeWindow() {
    ipcRenderer.send('window:minimize')
  },
})
```

@tab renderer.js

```js
const btn = document.getElementById('btn')

btn.addEventListener('click', () => {
  window.electronAPI.minimizeWindow()
})
```

:::

上面这类场景里，渲染进程并不关心“主进程返回什么结果”，它只是触发一个动作，因此 `send/on` 足够直接。

### `invoke/handle`：更适合请求-响应型消息

当渲染进程不仅要触发动作，还要拿到结果时，`invoke/handle` 会更自然。

系统对话框、文件读取、配置查询，本质上都属于这一类。

:::code-tabs

@tab main.js

```js
const { app, BrowserWindow, ipcMain, dialog } = require('electron')
const path = require('node:path')

function createWindow() {
  const win = new BrowserWindow({
    width: 800,
    height: 600,
    webPreferences: {
      preload: path.join(__dirname, 'preload.js'),
    },
  })

  win.loadFile('index.html')
}

ipcMain.handle('dialog:openFile', async () => {
  const focusedWindow = BrowserWindow.getFocusedWindow()
  return dialog.showOpenDialog(focusedWindow, {
    title: '请选择文件',
    properties: ['openFile'],
  })
})

app.whenReady().then(createWindow)
```

@tab preload.js

```js
const { contextBridge, ipcRenderer } = require('electron')

contextBridge.exposeInMainWorld('electronAPI', {
  openFileDialog() {
    return ipcRenderer.invoke('dialog:openFile')
  },
})
```

@tab renderer.js

```js
const btn = document.getElementById('open-btn')

btn.addEventListener('click', async () => {
  const result = await window.electronAPI.openFileDialog()
  console.log(result.filePaths)
})
```

:::

这里用 `invoke/handle` 的关键价值在于：==渲染进程拿到的是一个 Promise==。这让调用方式非常接近普通异步函数，阅读成本也更低。

### 频道设计为什么要重视

初学时最容易把 IPC 写成：

- `send('abc')`
- `send('test')`
- `send('event')`

这种写法短期能跑，长期一定难维护。因为频道名本质上就是跨进程协议的一部分。

:::details 频道命名建议

推荐把频道名写成“能力域:动作”的形式，例如：

- `dialog:openFile`
- `window:minimize`
- `store:get`
- `store:set`
- `app:getVersion`

这样做的好处是：

- 一眼看出这个消息属于哪个模块
- 不容易发生频道冲突
- 后面做权限收口时也更容易

:::

## 渲染进程间的通信

Electron 不提供“窗口 A 直接调用窗口 B 方法”这种模型。因为每个窗口本质上都是独立渲染进程，它们之间更合理的方式仍然是==通过主进程做中转==。

为什么需要中转？因为主进程天然掌握两个关键能力：

- 它知道当前有哪些窗口存在
- 它可以拿到每个窗口的 `webContents`

所以，渲染进程之间通信最常见的模型是：

```text
渲染进程 A -> 主进程 -> 渲染进程 B
```

### 主进程为什么适合作为中转站

课程代码里用一个对象记录了“某个频道有哪些窗口订阅”，整体思路是对的：

- 渲染进程先向主进程注册自己要监听的频道
- 主进程维护频道和窗口 `id` 的映射关系
- 需要广播或转发时，再根据 `webContents.id` 找到对应窗口并发送消息

这套模型解决的问题是：==让窗口之间不需要彼此持有引用，只需要约定频道即可==。

:::code-tabs

@tab main.js

```js
const { app, BrowserWindow, ipcMain } = require('electron')
const path = require('node:path')
const url = require('node:url')

const winRef = []
const messageChannelRecord = {}

function registerChannel(channel, webContentsId) {
  if (!messageChannelRecord[channel]) {
    messageChannelRecord[channel] = []
  }

  const exists = messageChannelRecord[channel].includes(webContentsId)
  if (!exists) {
    messageChannelRecord[channel].push(webContentsId)
  }
}

function getWebContentsIds(channel) {
  return messageChannelRecord[channel] || []
}

function transText(webContentsIds, channel, data) {
  for (let i = 0; i < webContentsIds.length; i++) {
    for (let j = 0; j < winRef.length; j++) {
      if (winRef[j].webContents.id === webContentsIds[i]) {
        winRef[j].webContents.send(channel, data)
        break
      }
    }
  }
}

ipcMain.on('registerChannelEvent', (event, channel) => {
  registerChannel(channel, event.sender.id)
})

ipcMain.on('transTextEvent', (event, channel, data) => {
  transText(getWebContentsIds(channel), channel, data)
})

function createWindow(page) {
  const win = new BrowserWindow({
    width: 600,
    height: 400,
    webPreferences: {
      nodeIntegration: true,
      contextIsolation: false,
    },
  })

  const pageUrl = url.format({
    protocol: 'file',
    pathname: path.join(__dirname, page),
  })

  win.loadURL(pageUrl)
  winRef.push(win)
}

app.whenReady().then(() => {
  createWindow('window1/window1.html')
  createWindow('window2/window2.html')
})
```

@tab renderer-a.js

```js
const { ipcRenderer } = require('electron')

ipcRenderer.send('registerChannelEvent', 'chat:message')

function sendMessage(text) {
  ipcRenderer.send('transTextEvent', 'chat:message', text)
}
```

@tab renderer-b.js

```js
const { ipcRenderer } = require('electron')

ipcRenderer.send('registerChannelEvent', 'chat:message')

ipcRenderer.on('chat:message', (_, text) => {
  console.log('收到另一个窗口的消息:', text)
})
```

:::

### 这种方案解决了什么问题

这类“主进程中转”方案，本质上解决了三个问题：

:::table full-width

| 问题 | 为什么会出现 | 中转方案怎么解决 |
| --- | --- | --- |
| 窗口彼此独立 | 每个窗口都是独立渲染进程 | 统一由主进程维护窗口引用 |
| 发送目标不明确 | 窗口之间不知道彼此 `webContents` | 主进程按频道维护订阅关系 |
| 广播和点对点都需要 | 有时给一个窗口，有时给多个窗口 | 主进程可以统一决定转发策略 |

:::

### 使用时要注意什么

这份课堂代码的思路是可用的，但如果要写进长期项目，建议补两点：

- 窗口关闭时要同步移除对应 `webContents.id`
- 频道注册表建议使用 `Map<string, Set<number>>`，比普通对象 + 数组更稳

:::warning 代码里有一个明显循环错误
课件代码的 `transText` 里第二层循环写成了 `for (let j = 0; i < winRef.length; j++)`，条件变量应当是 `j` 而不是 `i`。如果直接照抄，会导致转发逻辑异常。
:::

## 消息端口通信

前面两节的 IPC 更像“按频道发送事件”。但有些场景里，我们需要的是==建立一条持续可双向读写的通道==，这时消息端口会更合适。

### `MessageChannel` 解决了什么问题

`MessageChannel` 是浏览器里的 Web API，它会创建一对端口：

- `port1`
- `port2`

两个端口天然互通。你可以把它理解成一根双向管道，两边只需要持有各自的端口，就可以不停收发消息。

它特别适合：

- 主线程与 `Worker` 通信
- 两个隔离上下文之间建立专属通道
- 不希望把所有消息都塞进全局 IPC 频道时

### 渲染进程中的 `MessageChannel`

在渲染进程中，因为本质上还是 Chromium 环境，所以可以直接使用原生 `MessageChannel`。

课程代码用的是“主线程 + Worker”通信，这个例子很适合帮助理解消息端口的工作方式。

:::code-tabs

@tab main.js

```js
// 1. 先启动一个 Worker
// 2. 再创建一条 MessageChannel
// 3. 自己保留 port1，把 port2 交给 Worker
document.getElementById('sendMessage').addEventListener('click', () => {
  const worker = new Worker('worker.js')
  const channel = new MessageChannel()

  channel.port1.onmessage = (event) => {
    document.getElementById('data-display').textContent = `从 Worker 接收到的数据：${event.data}`
  }

  worker.postMessage('Hello World', [channel.port2])
})
```

@tab worker.js

```js
self.onmessage = (event) => {
  const port = event.ports[0]

  if (port) {
    setInterval(() => {
      port.postMessage(Math.random())
    }, 1000)
  }
}
```

:::

这个例子里真正要理解的是“端口转移”这件事：

- 页面创建通道
- 页面留下 `port1`
- 页面把 `port2` 作为可转移对象交给 `Worker`
- 从此双方不需要再依赖外层上下文，直接用端口通信

### Electron 主进程里的对应能力

`MessageChannel` 本身是 Web API，主进程里不能直接用。但 Electron 给主进程补了一个对应实现：`MessageChannelMain`。

这意味着 Electron 提供了两套相似能力：

:::table full-width

| 环境 | 可用能力 | 说明 |
| --- | --- | --- |
| 渲染进程 | `MessageChannel` | 浏览器原生支持 |
| 主进程 | `MessageChannelMain` | Electron 专门补充 |

:::

当你需要主进程和渲染进程之间建立一条长期端口通信链路时，可以使用 `MessageChannelMain` 把端口转交给页面。

:::details 什么时候该考虑消息端口

如果你的通信只是：

- 打开文件
- 获取系统信息
- 让主进程执行某个动作

那么普通 IPC 就够了。

如果你的通信更像：

- 持续推送状态
- 高频率双向收发
- 想为某个上下文单独建立长期通道

这时再考虑消息端口更合适。

:::
