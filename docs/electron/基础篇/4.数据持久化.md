Electron 的数据持久化和普通前端最大的区别在于：它不是只能选浏览器能力，也不是只能选 Node.js 能力，而是两边都能用。

这带来一个结果：==选择很多，但也更容易乱==。

所以这章先解决一个更本质的问题：Electron 里到底有哪些持久化路径，它们分别适合什么场景。

## 选型总览

:::table full-width

| 方案 | 存储位置 | 适合存什么 | 优点 | 代价 |
| --- | --- | --- | --- | --- |
| `fs` / `fs-extra` | 本地文件系统 | 文本文件、导出文件、用户目录下的数据文件 | 灵活、可控、适合文件型数据 | 你要自己管路径和格式 |
| `electron-store` | `userData` 目录下的 JSON 文件 | 配置项、轻量级 JSON 数据 | API 简单、开箱即用 | 不适合复杂查询 |
| `localStorage` | 渲染进程浏览器存储 | 少量简单键值 | 上手快 | 容量小、能力弱 |
| `IndexedDB` | 渲染进程浏览器数据库 | 大量结构化数据 | 浏览器原生、支持索引 | 原生 API 偏底层 |
| `Dexie.js` | IndexedDB 的封装层 | 结构化业务数据 | API 更友好 | 需要额外依赖 |

:::

## 基于 Node.js 的本地文件持久化

如果你的数据天然就是“文件”，例如：

- 导出的文档
- 用户手动选择的保存位置
- 程序运行生成的本地配置

那么这类数据最稳的方式通常不是塞进浏览器存储，而是直接走文件系统。

### 为什么先关心存储目录

很多人上来就直接 `writeFileSync('./xxx')`，但桌面应用真正应该先解决的是：==写到哪里==。

安装目录并不可靠，因为它：

- 可能没有写权限
- 更新时可能被覆盖
- 不适合存用户个性化数据

Electron 提供了 `app.getPath(...)`，就是为了让你拿到对应操作系统推荐的目录。

### 常用路径

```js
console.log(app.getPath('userData'))
console.log(app.getPath('home'))
console.log(app.getPath('desktop'))
console.log(app.getPath('documents'))
console.log(app.getPath('downloads'))
```

其中最重要的是：

- `userData`：最适合应用自己的配置和用户数据
- `desktop` / `documents` / `downloads`：更适合用户主动导出文件

:::tip 一条经验
应用内部配置优先放 `userData`，用户明确感知的文件导出再放桌面、文档或下载目录。
:::

### 用户数据目录为什么默认就够用

因为各操作系统都有自己推荐的应用数据目录。Electron 已经帮你做了这一层适配，所以大多数情况下，直接使用 `app.getPath('userData')` 就足够合理。

### `fs` 和 `fs-extra` 的区别

课程里专门对比了原生 `fs` 和 `fs-extra`，这点很有价值。

原生 `fs` 的问题不是不能用，而是做一些常见文件操作时样板代码比较多，比如：

- 确保目录存在
- 递归删除目录
- 递归复制

`fs-extra` 本质上是在 `fs` 的基础上补了一层更适合工程使用的 API。

:::code-tabs

@tab fs.js

```js
const fs = require('fs')
const path = require('path')

const dirPath = path.join(__dirname, 'exampleDir')

if (!fs.existsSync(dirPath)) {
  fs.mkdirSync(dirPath, { recursive: true })
}
```

@tab fs-extra.js

```js
const fs = require('fs-extra')
const path = require('path')

const dirPath = path.join(__dirname, 'exampleDir')

fs.ensureDir(dirPath)
  .then(() => console.log('Directory ensured successfully!'))
  .catch(err => console.error(err))
```

:::

### 文件保存示例

下面这段课堂代码展示了两个典型场景：

- 直接保存到桌面
- 用户先选目录，再保存到指定目录

```js
const { app, ipcMain, dialog } = require('electron')
const fs = require('fs')
const path = require('path')

ipcMain.on('save-to-desktop', (_, text) => {
  const desktopPath = app.getPath('desktop')
  fs.writeFileSync(path.join(desktopPath, 'myTextFile.txt'), text)
})

ipcMain.handle('select-dir', async () => {
  const result = await dialog.showOpenDialog({
    properties: ['openDirectory'],
  })
  return result.filePaths[0]
})

ipcMain.on('save-to-dir', (_, { dirPath, text }) => {
  fs.writeFileSync(path.join(dirPath, 'myTextFile.txt'), text)
})
```

这段代码背后的设计思路很清楚：

- 主进程负责文件系统访问
- 渲染进程负责触发保存动作
- 用户可感知的保存路径，让用户自己选择

## `electron-store`：轻量级配置存储

如果你要存的不是“文件”，而是应用配置，例如：

- 最近打开的项目
- 窗口尺寸
- 用户偏好设置
- 小体量 JSON 数据

这时直接上文件系统当然也能做，但 `electron-store` 会更省心。

### 它解决了什么问题

它帮你省掉了三件事：

- 自己决定配置文件该放哪里
- 自己管理 JSON 序列化 / 反序列化
- 自己处理文件不存在时的初始化逻辑

```js
const Store = require('electron-store')

const store = new Store({
  name: 'my-first-electron-store-data',
})

store.set('name', 'xiejie')
console.log(store.get('name'))

store.set('foo.bar', 'this is a bar')
console.log(store.get('foo'))
console.log(store.get('aaa'))
```

### 常用 API

:::table full-width

| API | 作用 |
| --- | --- |
| `set(key, value)` | 写入数据 |
| `get(key, defaultValue?)` | 读取数据 |
| `has(key)` | 判断键是否存在 |
| `delete(key)` | 删除某个键 |

:::

### 它适合什么，不适合什么

适合：

- 配置
- 轻量状态
- 用户偏好

不适合：

- 大量结构化查询
- 复杂事务
- 高频、大体量业务数据

## 基于浏览器能力的持久化

当数据本身更贴近“页面状态”或“前端业务数据”时，浏览器侧持久化会更自然。

这类方案的核心特点是：它跟随渲染进程，而不是直接跟随主进程。

## `localStorage`

`localStorage` 的优势只有一个：简单。

但正因为简单，所以它更适合：

- 少量字符串配置
- 页面级状态缓存
- 一些不复杂的持久化偏好

### Electron 里的一个特殊点：多窗口共享

课程里有一个非常值得记住的点：==多个 `BrowserWindow` 默认可能共享同一个 `localStorage` 空间==。

这意味着：

- 你以为是窗口独立状态
- 实际上多个窗口可能读写同一份数据

如果不提前理解这一点，多窗口应用里很容易出现“怎么另一个窗口也读到了我的值”的困惑。

### `partition` 为什么重要

Electron 提供了 `partition` 来切分会话存储空间。

```js
const secondWin = new BrowserWindow({
  width: 800,
  height: 600,
  webPreferences: {
    nodeIntegration: true,
    contextIsolation: false,
    partition: 'persist:myCustomPartition',
  },
})
```

这里的含义是：

- `persist:`：这是一个持久会话
- `myCustomPartition`：这是这份存储空间的名字

也就是说，`partition` 解决的是==不同窗口是否共享同一份浏览器存储空间==的问题。

## IndexedDB

当你发现 `localStorage` 已经不够用了，通常意味着你需要真正的客户端数据库能力。

这时就该考虑 `IndexedDB`。

### 为什么需要它

因为它不是简单键值存储，而是支持：

- 对象仓库
- 索引
- 事务
- 大量结构化数据

所以它更像“浏览器里的本地数据库”。

### 它的核心概念

:::table full-width

| 概念 | 可以怎么理解 |
| --- | --- |
| 数据库 | 一个本地数据库实例 |
| 对象仓库 | 类似一张表 |
| 事务 | 一次读写操作的上下文 |
| 键 | 数据主键 |
| 索引 | 用来加速查询 |

:::

### 原生用法为什么显得啰嗦

因为它是低级 API。你必须自己处理：

- 打开数据库
- 版本升级
- 建仓库
- 事务生命周期
- 错误回调

```js
let db
const request = indexedDB.open('MyTestDatabase', 1)

request.onerror = function (event) {
  console.log('Database error:', event.target.errorCode)
}

request.onsuccess = function (event) {
  db = event.target.result
}

request.onupgradeneeded = function (event) {
  const database = event.target.result
  const objectStore = database.createObjectStore('name', { keyPath: 'id' })
  objectStore.createIndex('name', 'name', { unique: false })
}
```

后面增删改查，也都要依附事务来做。

### 它适合什么场景

如果你的 Electron 应用需要：

- 本地离线数据
- 大量列表数据
- 搜索和索引
- 比配置更重的业务状态

那 `IndexedDB` 会比 `electron-store` 更合适。

## Dexie.js

当你已经确定需要 `IndexedDB`，下一个问题通常是：是否要直接写原生 API？

如果项目规模稍微上来，我更倾向于直接上 `Dexie.js`。

### 它解决了什么问题

它不是替代数据库能力，而是替代原生 IndexedDB 那套==偏底层、样板多、回调重==的写法。

```js
let db = new Dexie('testDb')
db.version(1).stores({ articles: 'id', settings: 'id' })

await db.articles.add({ id: 0, title: 'test' })
await db.articles.filter(article => article.title === 'test')
await db.articles.put({ id: 0, title: 'testtest' })
await db.articles.delete(id)
await db.articles.orderBy('title')
```

### 为什么它对 Electron 很友好

因为 Electron 里大量“本地业务数据”的需求，本质上和离线 Web 应用很像。Dexie 让你既能保留 IndexedDB 的存储能力，又能用更工程化的方式组织代码。

### 什么时候该用 Dexie

如果你已经满足下面任意两条，就基本可以考虑它了：

- 数据量不小
- 需要按字段检索
- 需要排序 / 过滤
- 需要版本升级
- 不想手写一堆 IndexedDB 样板代码

## 一套更实用的选择逻辑

:::::steps

1. 先判断数据是不是“文件”

   如果用户会直接感知它是一个文件，例如导出的文本、日志、图片、配置文件，那么优先考虑 `fs` / `fs-extra`

2. 再判断是不是“轻量配置”

   如果只是主题设置、窗口状态、最近打开记录这类 JSON 配置，优先考虑 `electron-store`

3. 如果是页面侧结构化数据，再看复杂度

   小而简单：`localStorage`

   数据量大、需要查询：`IndexedDB` 或 `Dexie.js`

4. 多窗口场景下，一定先确认存储是否共享

   尤其是 `localStorage`，需要提前决定是否通过 `partition` 做隔离

:::::
