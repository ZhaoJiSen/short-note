桌面应用和普通网页最大的差异之一，就是它不仅操作页面，还要操作系统能力。打开文件、保存文件、显示菜单、弹出右键菜单，这些都不是浏览器页面的职责，而是 Electron 作为桌面壳层提供的能力。

所以这章的核心不是“怎么调一个 API”，而是理解：==为什么这些能力通常要从主进程统一暴露出去==。

## 系统对话框

文件选择器、目录选择器、保存面板，本质上都属于系统级 UI。它们之所以应该放在主进程处理，是因为：

- 主进程掌握原生窗口引用
- 主进程天然更适合调用系统模块
- 渲染进程只需要拿结果，不需要直接控制系统对话框

### 为什么这里特别适合 `invoke / handle`

系统对话框天然就是“请求-响应”模型：

- 渲染进程发起请求
- 主进程打开系统对话框
- 用户选择后返回结果

这和 `ipcRenderer.invoke` / `ipcMain.handle` 的语义完全一致。

:::code-tabs

@tab main.js

```js
const { app, BrowserWindow, ipcMain, dialog } = require('electron')

ipcMain.handle('show-open-dialog', async function () {
  const window = BrowserWindow.getFocusedWindow()

  return dialog.showOpenDialog(window, {
    title: '我要打开一个文件',
    buttonLabel: '点击该按钮打开文件',
    defaultPath: app.getPath('pictures'),
    properties: ['openFile', 'openDirectory', 'multiSelections'],
    filters: [
      { name: 'Images', extensions: ['jpg', 'png', 'gif'] },
      { name: 'Movies', extensions: ['mkv', 'avi', 'mp4'] },
    ],
  })
})
```

@tab renderer.js

```js
const { ipcRenderer } = require('electron')

const btn = document.getElementById('btn')

btn.addEventListener('click', async () => {
  const result = await ipcRenderer.invoke('show-open-dialog')
  console.log(result)
})
```

:::

### `BrowserWindow.getFocusedWindow()` 为什么常和对话框一起出现

因为系统对话框最好挂在“当前用户正在操作的窗口”上。

如果你不传窗口引用，很多时候对话框也能打开，但体验会变差：

- 对话框可能不是当前窗口的模态层
- 多窗口场景下，用户会疑惑“这个对话框属于谁”
- 焦点和层级会变得混乱

### 常见对话框参数

:::table full-width

| 选项 | 作用 | 常见用法 |
| --- | --- | --- |
| `title` | 对话框标题 | 提示当前操作 |
| `buttonLabel` | 确认按钮文本 | 提升可读性 |
| `defaultPath` | 默认打开目录 | 提升选择效率 |
| `properties` | 控制选择能力 | `openFile`、`openDirectory`、`multiSelections` |
| `filters` | 文件类型过滤 | 限制可选后缀 |

:::

## 菜单

菜单不是“可有可无的装饰”，而是桌面应用的重要交互入口。尤其是在 macOS 上，系统甚至默认期待应用有菜单栏。

Electron 提供了两个关键方法：

- `Menu.buildFromTemplate`
- `Menu.setApplicationMenu`

这意味着菜单不是写死的，而是通过一份模板配置描述出来的。

### 自定义菜单为什么是模板驱动

因为菜单天生具有层级结构：

- 顶层菜单
- 子菜单
- 分隔线
- 点击回调
- 快捷键
- 系统角色

把它抽成模板，比手动一项项创建更自然。

:::code-tabs

@tab menu.js

```js
const { Menu } = require('electron')

const menuArr = [
  {
    label: '编辑',
    submenu: [
      { label: '撤销', role: 'undo' },
      { label: '重做', role: 'redo' },
      { type: 'separator' },
      { label: '剪切', role: 'cut' },
      { label: '复制', role: 'copy' },
      { label: '粘贴', role: 'paste' },
      { label: '全选', role: 'selectall' },
    ],
  },
  {
    label: '菜单1',
    submenu: [
      { label: '菜单1-1' },
      {
        label: '菜单1-2',
        click() {
          console.log('你点击了菜单1-2')
        },
      },
    ],
  },
  {
    label: '开发者工具',
    submenu: [
      {
        label: '切换开发者工具',
        accelerator: process.platform === 'darwin' ? 'Alt+Command+I' : 'Ctrl+Shift+I',
        click(_, focusedWindow) {
          if (focusedWindow) focusedWindow.toggleDevTools()
        },
      },
    ],
  },
]

const menu = Menu.buildFromTemplate(menuArr)
Menu.setApplicationMenu(menu)
```

@tab index.js

```js
require('./menu')
```

:::

### `role` 为什么值得优先使用

如果某个菜单项本质上就是系统通用行为，例如：

- 复制
- 粘贴
- 剪切
- 撤销
- 重做

那么优先用 `role` 比自己写 `click` 更好。

原因有三个：

- Electron 会自动帮你接好系统行为
- 系统菜单体验更一致
- 一些状态会自动管理，例如无法粘贴时自动禁用

:::details 什么时候不该用 `role`

当一个菜单项是业务逻辑，例如：

- 打开最近项目
- 导出当前文档
- 切换某个业务模式

这时再使用 `click` 去写你自己的逻辑。

:::

### 为什么自定义菜单后要补开发者工具入口

课程里专门强调了这一点，这是因为 Electron 默认菜单本来就带有一些演示性质的开发能力。一旦你用自定义菜单覆盖掉默认菜单，这些快捷入口也没了。

所以，如果你还在开发阶段，最好主动补回：

- `toggleDevTools`
- 对应快捷键

否则调试体验会明显变差。

## 右键菜单

右键菜单有两种常见思路：

- 真正使用 Electron 原生菜单
- 在页面里自己绘制一个 DOM 菜单

课件示例展示的是第二种，也就是“网页式右键菜单”。

### DOM 版右键菜单解决了什么问题

它的优点在于：

- 完全由页面控制样式
- 易于和当前 UI 风格统一
- 不需要切到系统菜单模型

但它本质上仍然是页面层交互，不是真正的原生右键菜单。

```js
const menu = document.getElementById('menu')

window.oncontextmenu = function (e) {
  e.preventDefault()
  menu.style.left = `${e.clientX}px`
  menu.style.top = `${e.clientY}px`
  menu.style.display = 'block'
}

document.querySelector('.menu').onclick = function () {
  console.log('这是右键菜单上面的某一个功能')
}

window.onclick = function () {
  menu.style.display = 'none'
}
```

### 这个实现的关键控制点

:::table full-width

| 交互点 | 为什么必须处理 |
| --- | --- |
| `contextmenu` 事件 | 拦截浏览器默认右键菜单 |
| 菜单项点击事件 | 让菜单真正执行动作 |
| 点击空白区域隐藏 | 避免菜单一直残留 |

:::

### 什么时候选原生菜单，什么时候选 DOM 菜单

如果你更关心：

- 系统一致性
- 原生体验
- 快捷键和菜单角色支持

优先考虑 Electron 原生菜单。

如果你更关心：

- 完全自定义样式
- 更像网页应用的交互面板
- 只是局部业务菜单

DOM 版菜单会更灵活。
