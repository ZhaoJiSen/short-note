如果说 Electron 安全模型里有一个必须真正理解的概念，那就是：==不要让渲染进程直接拥有它不该拥有的能力==。

而“预加载脚本”和“上下文隔离”，就是围绕这个目标设计出来的。

这章的核心问题是：

- 页面为什么不能直接拿 Node.js 能力
- 预加载脚本为什么存在
- 在开启上下文隔离后，应该怎样安全地暴露 API

## 预加载脚本

所谓预加载脚本（preload script），指的是：==运行在渲染进程里，但会先于页面脚本执行的一段代码==。

它的位置很特殊：

- 它属于渲染进程侧
- 但它能接触更多底层能力
- 它天然适合作为“安全代理层”

### 为什么 Electron 要多这一层

因为页面并不应该直接接触：

- Node.js 文件系统 API
- Electron 主进程能力
- 一些高权限桌面操作

但页面又确实需要使用这些能力的一部分。于是就需要一层“筛选后的桥”来做中介，这层桥就是 preload。

### 最原始的 preload 思路

在未开启上下文隔离、或者你手动关闭了它时，preload 和页面共享同一个 `window`，因此可以直接挂载属性。

```js
// preload.js
const fs = require('fs')

window.myAPI = {
  write: fs.writeSync,
}
```

主进程里通过 `webPreferences.preload` 指定它：

```js
webPreferences: {
  nodeIntegration: true,
  contextIsolation: false,
  preload: path.join(__dirname, 'preload.js'),
}
```

这种写法在早期教程里很常见，但它有一个前提：==上下文没有隔离==。

## 上下文隔离

从 Electron 12 开始，上下文隔离默认就是开启的。它的目的非常明确：==把 preload 所在的受信任环境，和页面 JS 所在的环境隔离开==。

### 它解决了什么问题

如果没有上下文隔离，那么页面脚本和 preload 实际上共享一个全局环境。这样一来：

- 页面更容易污染全局对象
- 恶意脚本更容易碰到高权限能力
- 预加载层很难真正扮演“安全边界”

开启之后，页面侧拿不到 Node.js 运行环境，也无法直接访问 preload 里的局部变量。

:::tip 一句话理解
preload 是“受信任桥层”，页面是“低权限消费层”，上下文隔离就是强制把这两层拆开。
:::

## `contextBridge`：隔离之后怎样暴露能力

既然开启隔离之后，preload 不能再直接改页面的 `window`，那么如何把能力安全暴露给渲染进程？

答案就是 `contextBridge`。

它的思路不是“把整个 Node.js 给页面”，而是：==只把你明确挑出来的、受控的 API 暴露出去==。

### `exposeInMainWorld`

这是最常用的方法。它的语义非常清楚：把一个 API 对象挂到页面主世界的 `window` 上。

```js
const fs = require('fs')
const { contextBridge } = require('electron')

contextBridge.exposeInMainWorld('myAPI', {
  write: fs.writeSync,
  open: fs.openSync,
})
```

渲染进程里再这样使用：

```js
console.log(window.myAPI)

const btn = document.getElementById('btn')
btn.addEventListener('click', () => {
  const fd = window.myAPI.open('example.txt', 'w')
  window.myAPI.write(fd, 'This is a test')
})
```

### 为什么这种方式比直接开 `nodeIntegration` 更安全

因为页面只能使用你明确暴露出来的方法，而不是整个 `fs`、整个 `electron`、整个 Node.js 环境。

这就意味着：

- 暴露面更小
- 权限边界更清楚
- 后续审查和维护更容易

## 一套更完整的示例

课程代码里已经把 preload 和 `contextBridge` 用起来了。下面整理成更清晰的三段式结构：主进程、预加载脚本、渲染进程。

:::code-tabs

@tab main.js

```js
const { app, BrowserWindow } = require('electron')
const path = require('node:path')

function createWindow() {
  const win = new BrowserWindow({
    width: 800,
    height: 600,
    webPreferences: {
      preload: path.join(__dirname, 'preload.js'),
    },
  })

  win.loadFile('window/index.html')
}

app.whenReady().then(createWindow)
```

@tab preload.js

```js
const fs = require('fs')
const { contextBridge } = require('electron')

contextBridge.exposeInMainWorld('myAPI', {
  write: fs.writeSync,
  open: fs.openSync,
})
```

@tab renderer.js

```js
const btn = document.getElementById('btn')

btn.addEventListener('click', () => {
  const fd = window.myAPI.open('example.txt', 'w')
  window.myAPI.write(fd, 'This is a test')
})
```

:::

### 这个例子真正想说明什么

不是“页面终于又能直接写文件了”，而是：

- 页面没有直接得到 `fs`
- 页面只是拿到了 preload 精选后暴露出的两个方法
- 权限的入口被集中到了 preload

这就是 Electron 推荐的安全模型。

## `exposeInIsolatedWorld`

除了 `exposeInMainWorld`，`contextBridge` 还提供了 `exposeInIsolatedWorld`。它适合更细粒度的隔离场景，也就是把 API 暴露到某个指定的隔离世界，而不是主世界。

不过对绝大多数业务应用来说，最常用、也最容易维护的仍然是 `exposeInMainWorld`。

:::details 什么时候需要考虑 `exposeInIsolatedWorld`

只有当你明确在维护多世界脚本注入模型，例如：

- 插件系统
- 更复杂的隔离沙箱
- 需要区分多个脚本执行世界

这时才有必要进一步研究它。

:::

## 实战里真正要避免的写法

下面几种写法，虽然能跑，但长期都不建议：

:::table full-width

| 写法 | 问题 |
| --- | --- |
| 开启 `nodeIntegration` 让页面直接 `require('fs')` | 页面权限过大，安全边界消失 |
| preload 里直接把整个 `fs` 暴露给页面 | 暴露面过大，后续难审计 |
| preload 和页面通过共享全局对象通信 | 一旦上下文隔离开启，模型就不成立 |
| 把主进程对象直接暴露给页面 | 重新走回 `remote` 的老路 |

:::

## 一条更稳的设计原则

:::::steps

1. 页面先声明“我需要什么能力”

   例如：选择文件、读取配置、保存文本

2. preload 只暴露最小 API 面

   不要把底层模块整个暴露出去，只暴露业务真正需要的方法

3. 真正高权限操作尽量落回主进程

   特别是文件系统、系统对话框、窗口控制这类能力，更适合通过 IPC 让主进程执行

4. 把 preload 当成“安全边界层”而不是“方便层”

   它不是为了让你少写代码，而是为了让权限控制更明确

:::::
