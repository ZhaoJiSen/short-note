桌面应用的“桌面感”，很大一部分来自窗口系统。网页更关注页面布局，而 Electron 应用必须真正管理窗口：窗口大小、位置、层级、父子关系、多窗口之间的协作，这些都属于主进程要负责的事情。

所以这章的重点不是“会创建一个窗口”，而是理解两个问题：

- `BrowserWindow` 到底控制了哪些能力
- 当应用里出现多个窗口时，怎么让它们保持可管理

## 基本知识

Electron 中所有桌面窗口几乎都来自 `BrowserWindow`。它不是简单的“把一个 HTML 打开”，而是把一个原生窗口和一个渲染进程绑在一起。

### 为什么窗口配置值得单独学习

因为窗口配置直接决定了用户体验：

- 窗口能不能拖动和缩放
- 默认出现在哪个位置
- 标题栏和边框是否显示
- 是否始终置顶
- 是否是另一个窗口的子窗口

这些能力都不属于页面层，而属于主进程控制的桌面外壳。

### 常用配置项

:::table full-width

| 配置项 | 作用 | 典型场景 |
| --- | --- | --- |
| `width` / `height` | 初始宽高 | 定义窗口默认尺寸 |
| `minWidth` / `minHeight` | 最小尺寸 | 避免布局被压坏 |
| `maxWidth` / `maxHeight` | 最大尺寸 | 限制用户拖得过大 |
| `resizable` | 是否允许缩放 | 工具类弹窗常关闭 |
| `movable` | 是否允许移动 | 一些固定面板场景 |
| `x` / `y` | 初始位置 | 控制窗口出现坐标 |
| `title` | 标题栏文字 | 业务窗口命名 |
| `icon` | 标题栏图标 | Windows 桌面应用常见 |
| `frame` | 是否显示原生边框和标题栏 | 自定义标题栏时常设为 `false` |
| `parent` | 指定父窗口 | 父子窗口联动 |
| `alwaysOnTop` | 是否置顶 | 截图工具、悬浮面板 |

:::

### 标题栏文字为什么有优先级

窗口标题并不一定只由 `BrowserWindow.title` 决定。实际上它有一套优先级：

1. HTML 文档里的 `<title>`
2. `BrowserWindow` 配置中的 `title`
3. `package.json` 的 `name`
4. Electron 默认值

这件事看起来是细节，但实战里很容易踩坑：你明明在主进程里写了 `title`，结果页面真正显示出来的却是 HTML 文档标题。

### 基础示例

下面这段代码基本覆盖了课程里提到的常用配置：

```js
const { app, BrowserWindow } = require('electron')
const path = require('node:path')

function createWindow() {
  const win = new BrowserWindow({
    width: 600,
    height: 400,
    minWidth: 300,
    minHeight: 200,
    maxWidth: 900,
    maxHeight: 700,
    resizable: true,
    x: 100,
    y: 120,
    title: '主窗口',
    icon: path.join(__dirname, 'logo.ico'),
    webPreferences: {
      preload: path.join(__dirname, 'preload.js'),
    },
  })

  win.loadFile('index.html')
}

app.whenReady().then(createWindow)
```

### 父子窗口为什么不是“普通多窗口”

课程里专门提了“组合窗口”，这其实是在讲==窗口之间的关系==。

普通多窗口之间是独立的，但父子窗口会带来一组联动行为：

- 子窗口可以相对父窗口定位
- 父窗口移动时，子窗口跟着动
- 父窗口关闭，子窗口也会被销毁

:::code-tabs

@tab main.js

```js
const { app, BrowserWindow } = require('electron')

let parentWin = null
let childWin = null

function createWindow(config, parent) {
  const win = new BrowserWindow({
    width: config.width,
    height: config.height,
    parent: parent ?? null,
    webPreferences: {
      nodeIntegration: true,
      contextIsolation: false,
    },
  })

  win.loadFile(config.file)
  return win
}

app.whenReady().then(() => {
  parentWin = createWindow({ width: 600, height: 400, file: 'window/index.html' })
  childWin = createWindow({ width: 400, height: 200, file: 'window2/index.html' }, parentWin)

  const { x, y, width } = parentWin.getBounds()
  childWin.setPosition(x + width + 15, y)
  childWin.show()
})
```

@tab 说明

```text
这个例子里，childWin 被创建成了 parentWin 的子窗口。
所以它不只是“第二个窗口”，而是和父窗口建立了联动关系。
```

:::

### 窗口置顶解决什么问题

当应用存在多个窗口时，默认是后创建的窗口更靠上，但用户一点击别的窗口，层级又会重新变化。

有些工具类窗口并不希望被压住，例如：

- 小型控制面板
- 截图工具悬浮窗
- 临时确认框

这时就需要置顶。

```js
win.setAlwaysOnTop(true, 'pop-up-menu')
```

相比直接使用 `alwaysOnTop: true`，`setAlwaysOnTop(flag, level)` 的优势在于==粒度更细==，你可以控制它是普通置顶，还是更强层级的置顶。

## 多窗口管理

窗口一多，真正的问题不是“怎么再创建一个窗口”，而是==如何管理这些窗口的引用和状态==。

这也是课程里第 10 节的重点：窗口管理本质上是“引用管理”。

### 为什么要存窗口引用

因为 Electron 的窗口对象如果没有引用，后续你就很难再对它做这些操作：

- 再次显示
- 隐藏
- 关闭
- 按名字查找
- 分组广播

所以多窗口管理第一步几乎总是：==把窗口对象存起来==。

### 单窗口注册表：按名称管理

最简单的方式就是 `Map<string, BrowserWindow>`。

:::code-tabs

@tab main.js

```js
const { app, BrowserWindow, ipcMain } = require('electron')

const winMap = new Map()

function createWindow(config) {
  const win = new BrowserWindow({
    width: config.width,
    height: config.height,
    show: config.show,
    webPreferences: {
      nodeIntegration: true,
      contextIsolation: false,
    },
  })

  win.loadFile(config.file)
  winMap.set(config.name, win)

  if (config.name === 'win2') {
    win.on('close', (e) => {
      e.preventDefault()
      win.hide()
    })
  }
}

ipcMain.on('openWindow', (_, name) => {
  winMap.get(name)?.show()
})

ipcMain.on('closeWindow', (_, name) => {
  winMap.get(name)?.hide()
})

app.whenReady().then(() => {
  createWindow({ name: 'win1', width: 600, height: 400, show: true, file: 'window/index.html' })
  createWindow({ name: 'win2', width: 400, height: 200, show: false, file: 'window2/index.html' })
})
```

@tab 理解

```text
这里关闭 win2 时没有真的销毁，而是 hide。
这样做的目的，是下次再次打开时可以秒开，不必重新创建渲染进程。
```

:::

### 分组管理：按业务域管理多个窗口

当窗口不仅有多个，还天然属于不同业务分组时，简单的 “name -> window” 就不够了。

课程里的做法是：把 `Map` 的值变成数组，也就是：

```text
groupName -> BrowserWindow[]
```

这样之后就可以做到：

- 一个操作影响同一分组下的所有窗口
- 窗口关闭时，从分组里移除自己
- 分组为空时，删除该组

```js
const winMap = new Map()

function createWindow(config) {
  const win = new BrowserWindow({
    width: config.width,
    height: config.height,
    webPreferences: {
      nodeIntegration: true,
      contextIsolation: false,
    },
  })

  win.loadFile(config.file)

  if (config.group) {
    let groupArr = winMap.get(config.group)
    if (groupArr) {
      groupArr.push(win)
    } else {
      groupArr = [win]
    }
    winMap.set(config.group, groupArr)

    win.on('close', () => {
      groupArr = winMap.get(config.group)
      groupArr = groupArr.filter((item) => item !== win)
      winMap.set(config.group, groupArr)

      if (groupArr.length === 0) {
        winMap.delete(config.group)
      }
    })
  }

  return win
}
```

### 多窗口管理时真正要避免什么问题

很多人会把“窗口管理”理解成只是把窗口存进数组，但真正需要避免的是这三类混乱：

:::table full-width

| 问题 | 表现 | 后果 |
| --- | --- | --- |
| 引用丢失 | 创建后不保存窗口实例 | 后续无法 show / hide / close |
| 僵尸引用 | 窗口销毁后仍留在集合中 | 管理表越来越脏 |
| 创建与显示逻辑混在一起 | 每次打开都重新创建窗口 | 用户体验差，资源浪费 |

:::

所以更稳的实践是：

- 先决定窗口是“销毁”还是“隐藏复用”
- 再决定窗口是“按名称管理”还是“按分组管理”
- 最后统一在主进程收口这些操作
