Electron 应用不是“点一下页面脚本就跑起来”这么简单，它有自己的应用生命周期，也有窗口生命周期。什么时候初始化，什么时候创建窗口，什么时候在所有窗口关闭后退出，这些都属于生命周期管理。

如果对生命周期没有整体认识，最常见的结果就是：

- 初始化逻辑放错位置
- 多平台行为不一致
- 退出时清理不完整

## 应用生命周期

Electron 里最先接触到的生命周期几乎都是 `app` 模块提供的。

### 为什么 `whenReady` 总是最先出现

因为它就是应用真正“准备完成”的分界线。只有在 Electron 初始化完成之后，很多主进程 API 才适合开始使用。

```js
app.whenReady().then(() => {
  createWindow()
})
```

这段代码看起来普通，但它其实在表达一个非常重要的约束：==不要在应用尚未 ready 时就急着创建窗口或调用依赖环境初始化的能力==。

### 常见生命周期事件

:::table full-width

| 生命周期事件 | 触发时机 | 常见用途 |
| --- | --- | --- |
| `will-finish-launching` | 应用即将完成基础启动 | 某些平台级早期初始化 |
| `ready` | Electron 初始化完成 | 创建窗口、注册能力 |
| `window-all-closed` | 所有窗口都关闭时 | 控制是否退出应用 |
| `before-quit` | 退出前 | 保存状态、做最后清理 |
| `will-quit` | 即将退出 | 注销资源、解绑监听 |
| `quit` | 应用已退出 | 收尾日志 |

:::

### 为什么 `window-all-closed` 要区分平台

这是 Electron 生命周期里最典型的平台差异之一。

在 Windows / Linux 上，所有窗口关闭，通常意味着应用退出。  
但在 macOS 上，关闭所有窗口后应用常常仍然保持激活状态，用户可以再从 Dock 重新打开窗口。

```js
app.on('window-all-closed', () => {
  if (process.platform !== 'darwin') {
    app.quit()
  }
})
```

所以这段判断不是模板化“必须背下来”的代码，而是在对齐桌面系统的交互习惯。

## 窗口生命周期

除了 `app`，窗口本身也有一组生命周期事件。它们更贴近用户交互过程。

:::table full-width

| 事件 | 含义 | 常见用途 |
| --- | --- | --- |
| `closed` | 窗口已关闭 | 回收引用 |
| `focus` | 窗口获得焦点 | 更新激活状态 |
| `show` | 窗口显示 | 首屏埋点、恢复状态 |
| `hide` | 窗口隐藏 | 做后台挂起处理 |
| `maximize` | 窗口最大化 | 记录窗口状态 |
| `minimize` | 窗口最小化 | 托盘应用常用 |

:::

```js
win.on('minimize', () => {
  console.log('窗口最小化了')
})
```

### 为什么窗口事件值得单独监听

因为很多桌面应用行为不是页面事件能感知的。例如：

- 用户点了系统最小化按钮
- 窗口被隐藏到托盘
- 窗口被操作系统重新聚焦

这些都必须在 `BrowserWindow` 这一层处理。

## `remote` 模块为什么被废弃

课程资料把 `remote` 单独拿出来讲，这是对的。因为它曾经是 Electron 入门里最“诱人”的能力：渲染进程可以像直接拿到主进程对象一样去调用 `BrowserWindow`、`dialog` 等 API。

但问题也恰恰出在这里。

### 它看起来解决了什么问题

它试图解决的是：==让渲染进程更方便地访问主进程能力==。

例如过去可以这样做：

```html
<script>
  let { remote } = require('electron')
  let newWin = null

  btn.onclick = function () {
    newWin = new remote.BrowserWindow({
      webPreferences: { nodeIntegration: true }
    })
    newWin.loadFile('./index2.html')
  }
</script>
```

这类代码非常短，也很容易让人误以为“这才是最省事的方式”。

### 它为什么被弃用

真正的问题主要有三类：

:::table full-width

| 问题 | 说明 |
| --- | --- |
| 性能问题 | `remote` 的每次访问本质上都是跨进程调用，代价远高于本地调用 |
| 影子对象问题 | 渲染进程拿到的并不是真正对象，而是代理对象，行为和原对象可能有差异 |
| 安全问题 | 渲染进程过于轻易地接触主进程能力，扩大攻击面 |

:::

### 版本演进

- Electron 10 起：`remote` 需要手动开启
- Electron 14 起：官方彻底废弃 `remote`

如果历史项目里还在使用它，现在更合理的替代路径是：

- 用 `IPC`
- 用 `preload + contextBridge`
- 必要时使用 `@electron/remote`，但要明确知道这是兼容方案，不是推荐新方案

:::warning 不要把 `remote` 当作“写起来更方便”的捷径
它绕过的，其实正是 Electron 想让你显式管理的进程边界。短期省事，长期一定会在性能和安全上还债。
:::
